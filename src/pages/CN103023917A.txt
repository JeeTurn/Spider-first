    <div class="abstract">本发明提出一种针对智能家电进行授权的方法，包括：平台服务器接收智能家电根据用户指令发送的第一请求消息；平台服务器根据第一请求消息生成标识码，并根据标识码和平台服务器的地址生成对应的二维码；平台服务器将二维码发送至智能家电，并通过智能家电将二维码显示给用户；平台服务器接收用户通过移动终端拍摄并解析二维码之后生成的针对地址的第二请求消息；平台服务器根据第二请求消息对用户进行身份验证，并在用户通过身份验证之后，对智能家电进行授权。本发明不需要智能家电提供输入设备和浏览器便可访问平台服务器数据，过程简单，易于操作，用户体验好。本发明还公开了一种针对智能家电进行授权的系统、一种智能家电和一种移动终端。</div>

    <p>针对智能家电进行授权的方法、系统和装置</p>
    <p>技术领域</p>
    <p>[0001]	本发明涉及互联网技术领域，特别涉及一种针对智能家电进行授权的方法、系统和装置。</p>
    <p>背景技术</p>
    <p>[0002]	智能家电将计算机技术引入家电设备，使得家电在具有自身电器功能之后，还具备了一定的计算机的能力，如互联网访问等。</p>
    <p>[0003]	出于用户数据安全性、隐私性等的考虑，目前所有的开放平台都会在向第三方开放用户相关数据前，通过一个登录授权界面征询该用户是否同意将该数据的相应访问权限授予给该第三方。只有当用户以自己在平台方的账号登录并同意将权限授予第三方之后， 第三方才能通过平台方提供的开放应用程序接口访问用户的相关数据。</p>
    <p>[0004]	在整个过程中，有两个非常关键的步骤：</p>
    <p>[0005]	第一：用户需要用其在平台方的账号登录平台方账户系统，这要求用户当前所使用的设备必须能够支持内容输入。一般需要具有外设键盘（如PC机），或内置按键（如使用键盘的手机），或触摸屏（如iphone、触屏智能手机及各种平板电脑等）。</p>
    <p>[0006]	第二 ：登录、授权界面一般是由平台方所提供的网页来进行显示，网页中一般会用到JS、cookie等前端技术，这些技术要求用户当前所使用的设备必须具有一个支持这些技术的浏览器。</p>
    <p>[0007]	然而，目前的大多数智能家电都不具备用户体验良好的输入设备，因此无法很好地帮助用户输入账号和密码。此外，目前大多数智能家电，尤其是早先推出的智能电视，不具备或者无法安装全功能的浏览器。随着Android4. O的推出，一些智能电视也开始内置了全功能浏览器，但输入设备仍基于电视遥控器，输入体验差，使用不方便。</p>
    <p>[0008]	现有的一些开放平台，如谷歌提供了供无良好输入设备或不具备全功能浏览器的设备的解决方案，基本使用流程如下：</p>
    <p>[0009]	步骤S1:智能电视向平台服务器发起一次网络请求，获得平台服务器分配的一个user code和device code,分别用于唯一标识用户和设备。</p>
    <p>[0010]	步骤S2 :智能电视将user code显示给用户，并提示用户打开一个指定的网页,用户可以选择在台式电脑、笔记本电脑、平板电脑、或智能手机的浏览器中打开网页。用户将看到的user code按照网页内容的提示输入并进行提交，然后通过轮询的方式不断向平台服务器发送device code数据以获取访问授权码。其中，用户必须先登录平台服务器，才能提交 user code。</p>
    <p>[0011]	步骤S3 :平台服务器接收登录用户发送的user code,为该user code对应的信息做相应处理并激活device code。激活后，当再次接收到设备发过来的devicecode时,就会为该device code所对应的user code所对应的用户生成一个访问授权码,称为accesstoken,并将access token返回给智能电视。</p>
    <p>[0012]	步骤S4 :智能电视接收到access token,就可以调用平台方提供的开放应用程序接口访问用户的相关数据。</p>
    <p>[0013]	上述技术的主要缺点是用户必须手动地在另外的设备上打开浏览器，输入所提示的网址，并在加载出来的网页上输入设备所显示的user code, user code &#8212;般是随机字符串。这个过程的输入成本是比较高的，程序比较繁琐，用户体验不够友好。</p>
    <p>发明内容</p>
    <p>[0014]	本发明旨在至少解决现有技术中存在的技术问题之一。</p>
    <p>[0015]	为此，本发明的一个目的在于提出一种针对智能家电进行授权的方法。</p>
    <p>[0016]	本方法不需要智能家电提供输入设备和浏览器，充分利用了用户的移动终端的能力，授权过程更加简单方便，易于操作，在使得智能家电能够访问平台服务器数据的同时提供更好的用户体验。 [0017]	本发明的第二个目的在于提出一种针对智能家电进行授权的系统。</p>
    <p>[0018]	本发明的第三个目的在于提出一种智能家电。</p>

    <p>[0130]	在本发明的一个实施例中，还包括：数据项管理模块460用于以设备标识码为索引词并以用户标识码为值建立第一数据项，同时用户标识码为索引词并以数据访问权限列表为值建立第二数据项。</p>

    <p>[0135]	例如，智能家电向平台服务器发送请求从而调用平台方的相关开放接口。发送的请求消息还包括智能家电的标识信息和用户请求的数据访问权限列表，分别记做AK、SCOPES，智能家电的标识信息是内置系统应用提供方在平台方注册应用时所得到的应用的唯一标识。其中，第一请求消息中还包括智能家电的应用密钥，记做SK，用来对智能家电进行验证。第一接收模块410接收到请求后先对智能家电提供的AK/SK、SCOPES参数进行校验，确认有效后，生成模块420根据第一请求消息，使用随机算法生成标识码，包括两个唯一标识串，分别作为发送当前请求的智能家电的唯一标识码和使用该智能家电的用户的唯一标识码，记做DEVICE CODE ，记做USER CODE。根据平台服务器的地址，生成一个包含USERCODE参数信息的网络地址，以及该地址所对应的二维码，记做QR。数据项管理模块460建立以DEVICE CODE为索引词，以USER CODE为值的第一数据项；数据项管理模块460建立以USER CODE为索引词，以数据集合（SCOPES，AK)为值的第二数据项存入数据库或缓存系统中，并返回DEVICE CODE、QR等数据。智能家电接收到的发送模块430发送的数据后，将接收到的二维码进行显示。智能家电通过轮询或长连接技术向平台方服务端发起查询请求，请求中携带DEVICE CODE等数据。用户通过移动终端上扫描显示在智能家电上的二维码图片并解析后发送第二请求消息，请求二维码对应网址。第二接收模块440接收HTTP请求形式的第二请求消息。授权模块450通过第二请求消息中的cookie信息，与用户账号系统进行交互，从而判断出用户是否已经处于登录状态。如果用户已经处于登录状态，则从第二请求消息中取出USER CODE参数，并从数据库或缓存系统中读取以USER CODE为键的数据项，根据其中的SC0PES、AK数据以及USER CODE数据生成一个授权页面并返回至移动终端。如果判断用户尚未处于登录状态，则授权模块450返回一个重定向响应，将用户引导至登录页面。当用户完成登录后再重新加载二维码图片所编码的网络地址。移动终端通过例如浏览器加载授权页面，用户对所使用的智能家电的内置系统应用选择性地授予相应数据访问权限，并将授权请求发送至平台方服务器。授权模块450接收到用户从移动终端提交的授权请求，从请求包中提取出USER CODE,SCOPES的参数，并通过cookie信息及与用户账号系统的交互获得当前登录用户的UID数据。授权模块450将数据库或缓存系统中的以USERCODE为key的数据项更新为数据集合（SCOPES’，AK, UID)，将此数据集合作为用户的授权数据项，与更新比较，新增了 UID数据，且对应的SCOPES值可能会发生变化。授权模块450生成用户的授权数据项后，第一接收模块410接收到来自智能家电发起的轮询请求或长连接请求时，提取出请求中的DEVICE CODE参数，并以之为键从数据库或缓存系统中读取对应的USER CODE数据。如果对应的USER CODE数据不存在，则返回错误信息，否则则以USERCODE为键读取出用户的授权数据项（SCOPES’，AK, UID)。如果用户的授权数据项中不存在有效的UID数据项，则表明用户尚未授权，此时同样返回错误信息，否则则生成一个访问授权码，称为TOKEN，并返回给智能家电。</p>
    <p>[0136]	通过TOKEN可以获取当前授权用户的身份信息如WD、平台服务器分配给智能家电内置系统应用的应用认证信息如AK，以及用户授予给智能家电内置系统应用的数据访问权限列表如SCOPES等信息。智能家电接收到平台服务器返回的TOKEN数据就表示已经成功地获取到了用户的授权，通过将该TOKEN数据进行保存，并在之后的每次API调用时传入该TOKEN参数，就可以成功地访问到用户在平台服务器的相关数据。第三方服务端将用户绑定的其在第三方服务自有账号系统中的账号做自动登录处理，其后用户再以同样的账号登录时，第三方服务将会自动以用户所绑定的第三方服务账号系统中的账号登录，直到用户取消绑定或取消对该第三方服务的授权。</p>
