    <div class="abstract">本发明为一种移动查询系统的客户端以及前后台数据处理方法，所述的客户端包括：一功能管理单元，一数据操作单元，一通信线程处理单元，一界面处理单元以及一数据包处理单元，通过与中间平台建立通讯联系；由客户端的通信线程处理单元发出命令请求数据包，中间平台接到数据包并进行解析；从中间平台模版数据库中寻找与解析出的命令相匹配的业务请求，并向一业务服务平台发出请求；业务服务平台返回相应的应答数据包给中间平台；中间平台根据模版进行解析，并按照模版形成相应的数据格式返回客户端的数据包处理单元进行解包和解析处理。从而达到具备业务无关性和协议无关性，实现了所见即所得的瘦身C/S方式的目的。</div>

    <p>一种移动查询系统的客户端以及前后台数据处理方法</p>
    <p>技术领域</p>
    <p>本发明涉及的是一种移动查询系统，特别涉及的是一种移动查询系统的 客户端以及后台数据处理方法。</p>
    <p>背景技术</p>
    <p>对于现有技术而言C/S与B/S模式往往是构成系统软件客户端的结构，其中 C/S   (Client/Server,客户机/服务器）模式又称C/S结构，是软件系统体系结构 的一种。C/S模式简单地讲就是基于企业内部网络的应用系统。与B/S (Browser/Server,浏览器/服务器）模式相比，C/S模式的应用系统最大的好处是 不依赖企业外网环境，即无论企业是否能够上网，都不影响应用。</p>
    <p>C/S本身也是属于发展到90年代的较为先进的结构。这种模式将应用一分 为二，服务器负责数据管理，客户机完成与用户的交互任务。C/S具有强壮的数 据操纵和事务处理能力，以及数据的安全性和完整性约束。但随着企业规模的日 益扩大，应用程序的复杂程度不断提高，逐渐也暴露了以下几点缺陷：</p>
    <p>1、 开发成本较高。C/S结构对客户端软硬件要求较高，尤其是软件的不断升 级，对硬件要求不断提高，增加了整个系统的成本，客户端越来越臃肿；</p>
    <p>2、 移植困难。不同开发工具开发的应用程序， 一般来说互不兼容，不能搬到 其它平台上运行；</p>
    <p>3、 用户界面风格不一，使用繁杂，不利于推广使用；</p>
    <p>4、 维护复杂，升级麻烦，如果应用程序要升级，必须到现场为客户机&#8212;&#8212;升 级，每个客户机上的应用程序都需维护；</p>
    <p>5、 新技术不能轻易应用，因为一个软件平台及开发工具一旦选定，不可能轻 易更改。</p>
    <p>对于B/S而言，基于B/S的Intranet技术，其三项优势是实质性的： 1、它是有计算技术以来最稳定的技术平台，在C/S时代，用户最苦恼的问 题莫过于如何选择一个稳定的平台。由于以前的技术基础不是网络，因此自然地</p>
    <p>发展出众多的互不兼容的技术平台。每种技术平台，从用户应用稳定性要求来看， 都可以说是罢花一现，给用户留下了无奈与苦恼。只有进入网络计算机时代，由 于上亿用户都联在网上，其技术才获得了高度稳定性。它只能发展,不能被推翻， 所有人只能支持和遵循它,不能控制它；</p>
    <p>2、 它本质上是一种客户机技术，这对于大中型企业特别合适。在C/S模式下， 每种应用的主要部分是在客户机侧。别的不说，光是在众多的客户机侧安装和维 护多种应用的多个版本，就叫人受不了。而Internet技术却不同，上亿个用户只 是通过浏览器这个统一界面，就可享受到无限丰富的，永远在不断变化和发展着 的信息服务。Intranet原则上取消了所有在客户机侧的维护工作。</p>
    <p>3、 它提供了异种机、异种网、异种应用服务的联机、联网、统一服务的最现 实的开放性基础。通过Intranet技术统一访问异种数据库，早已成为现实。从ISDN 到ATM，再到现在大行其道的TCP/IP,正是由于Internet的大行其道，宽带IP 将会成为真正的统一服务的（数据、语音、多媒体）网络基础。但是存在着两大 问题，首先企业是一个有结构、有管理、有确定任务的有序实体，而Internet面 向的却是一个无序的集合，B/S必须适应并迎合长期C/S下的有序需求方式。其 次企业中已经积累了或多或少的各种基于非Internet技术上的应用，与这些应用 联接，是Intranet &#8212;项极其重要而繁重的任务。缺乏对动态页面的支持能力，没 有集成有效的数据库处理功能，系统的扩展能力差，安全性难以控制，好集成工 具不足等等，都让我们应慎重行事。最后，对于安全要求高的企业B/S模式存在 应用存在注入式漏洞及B/S高安全级别难于实现和控制。</p>
    <p>在现在的无限行业领域，国内开发无线应用平台的很多，但是采用瘦身C/S 方式的很少，能够真正实现中间平台业务无关的更少而能够实现所见即所得的瘦 身开发模式就基本上没有了，大多数&#37318;用传统的C/S方式；</p>
    <p>鉴于上述问题，本发明创作者以北京工商项目，这一目前国内最大的无线应 用项目为依托，经过长时间的开发终于获得本发明。</p>
    <p>发明内容</p>
    <p>本发明的目的在于，提供一种移动查询系统的客户端以及后台数据处理方法， &#37318;用先进的中间件技术，组件技术，具备业务无关性和协议无关性，实现了所见 即所得的瘦身C/S方式，用以克服上述缺陷。</p>
    <p>为实现上述目的，本发明采用的技术方案在于，首先提供一种移动查询系统 客户端，其包括： 一功能管理单元，其提供对各功能类型的管理操作，并完成功 能列表下载、模板自动升级、本地数据库的维护；</p>
    <p>一数据操作单元，其提供新建、修改、删除功能数据的操作处理；</p>
    <p>一通信线程处理单元，其提供对通信业务事件的响应处理，通过调用所述的 功能管理单元、数据操作单元，提供的功能接口完成各通信事务；</p>
    <p>一界面处理单元，其提供对图形界面事件的响应处理，同时也是WAC主应 用入口线程，其调用通信线程处理单元的接口函数启动通信线程；</p>
    <p>一数据包处理单元，其获得所述数据操作单元的数据进行通信协议打包，对 收到的数据包进行解包和解析处理，以及对通信连接进行维护；</p>
    <p>其中，所述的通信线程处理单元要实时进行检测，判断数据操作单元为在脱 机状态还是联机状态；</p>
    <p>其中，所述的数据操作单元包括：</p>
    <p>一脱机数据操作模块用以完成脱机状态下的新建、修改、删除功能数据的操 作处理；</p>
    <p>一联机数据操作模块用以完成联机状态下的新建、修改、删除功能数据的操 作处理，提供脱机数据的提交处理，其与所述的数据包处理单元相连接； 其中，所述的数据包处理单元包括：</p>
    <p>一协议处理模块用以提供对各种通信协议打包和解包的处理，对通信连接进 行维护；</p>
    <p>一 XML解析模块用以对收到的XML数据包进行解析处理； 其次提供一种移动查询系统客户端数据处理的方法，其是通过上述的移动查 询系统客户端实现的，其包括的步骤为：</p>
    <p>步骤a:通信线程处理单元进行线程的初始化； 步骤b:客户端发出操作指令，其包括：</p>
    <p>步骤M:界面处理单元产生一操作指令，将其传输给所述的功能管理单元； 步骤b2:所述的功能管理单元根据操作指令类型转化成相对应的功能数据； 步骤b3:判断客户端处于联机状态还是脱机状态，若是联机状态，执行下述 步骤b5,若是脱机状态，则执行下述步骤b4;</p>
    <p>步骤b4:由所述的脱机数据操作模块判断是对客户端自身数据进行的操作，</p>
    <p>若是则进行新建、修改、删除操作，然后执行下述步骤d，否则，将所述功能数</p>
    <p>据进行存储，执行上述步骤b3</p>
    <p>步骤b5:所述的联机数据操作模块获得所述功能数据，判断是对客户端自身 数据进行的操作，若是则进行新建、修改、删除操作，然后执行下述步骤d,否 则，并提交给所述的协议处理模块；</p>
    <p>步骤b6:所述的协议处理模块对数据进行打包处理，并由所述通信线程处理 单元发送出去；</p>
    <p>步骤c:客户端接到数据包，其包括：</p>
    <p>步骤cl:由所述的XML解析模块对接收到的XML数据包进行解析处理； 步骤c2:由所述的协议处理模块对通信协议进行解包处理； 步骤d:线程结束；</p>
    <p>其中，所述的功能数据为数据查询消息、数据修改消息、数据删除消息、选 择功能消息、登陆消息、注销消息、修改密码消息、更新登陆配置消息以及线程 退处消息；</p>
    <p>其中，还包括一步骤b41，设置于所述的步骤b4与步骤b5之间，所述的步</p>
    <p>骤b41:提示客户端没有与系统进行联机，处于脱机状态；</p>
    <p>最后提供一种移动查询系统后台数据处理方法，其包括的步骤有：</p>
    <p>一客户端的通信线程处理单元与 一 中间平台建立通讯联系；</p>
    <p>由所述客户端的通信线程处理单元发出命令请求数据包，所述中间平台接到</p>
    <p>数据包并进行解析；</p>
    <p>从所述中间平台模版数据库中寻找与解析出的命令相匹配的业务请求，并向 一业务服务平台发出请求；</p>
    <p>所述业务服务平台返回相应的应答数据包给所述中间平台；</p>
    <p>所述中间平台根据模版进行解析，并按照模版形成相应的数据格式返回客户 端的数据包处理单元进行解包和解析处理；</p>
    <p>还包括：在所述的中间平台根据模版对业务服务平台返回相应的应答数据包</p>
    <p>进行解析时，并将其记录到日志里；</p>
    <p>其中，在所述客户端与中间平台建立通讯联系之后，且在所述客户端发出命 令请求数据包之前，还包括一步骤：所述中间平台解析现有客户端的功能模版列 表与所述中间平台模版列表比较判断是否 一 致，若不 一 致则要将与中间平台模版</p>
    <p>列表相对应的功能模版下载到客户端。</p>
    <p>与现有技术比较本发明的优点在于：</p>
    <p>1、 采用瘦身C/S方式解决了普通C/S模式的开发成本高、界面风格不统一、 维护困难等缺点。采用瘦身C/S方式其界面由工具生成，犹如B/S的所见即所得 的方式，大大节约了开发界面成本；另外由同一后台配置，界面由后台生成，界 面统一；界面由后台生成，只要对后台进行维护，客户端进行下载即可完成维护， 大大减少维护费用。</p>
    <p>2、 本系统是专为无线应用而开发的，与B/S相比具有传输速度快、安全系数 高及能随时控制无线设备的优势。采用瘦身C/S方式，其本身就是基于C/S模式 延伸的，其网络传输无需生成复杂的页面，从而大大降低了网络消耗；并且具有 强壮的数据操纵和事务处理能力，以及数据的安全性和完整性约束，从而保证了 安全性；另为也避免了 B/S方式存在的无法调用无线设备及设备之间传递信息的 问题。</p>

    <p>本发明的其特点在于：</p>
    <p>1、 可扩展性：信息产业技术飞速发展，业务量的急速上升， 一年前的网络， 现在需要新一轮的扩展。合适的均衡解决方案应能满足这些需求，能均衡不同操 作系统和硬件平台之间的负载，不同服务器的负载，并且能以对客户端完全透明 的方式动态增加或删除某些资源。</p>
    <p>2、 灵活性：均衡解决方案应能灵活地提供不同的应用需求，满足应用需求的 不断变化。在不同的服务器群有不同的应用需求时，应有多样的均衡策略提供更 广泛的选择。</p>
    <p>3、可靠性：在对服务质量要求较高的站点，负载均衡解决方案应能为服务器 群提供完全的容错性和高可用性。但在负载均衡设备自身出现故障时，应该有良 好的冗余解决方案，提高可靠性。使用冗余时，处于同一个冗余单元的多个负载 均衡设备必须具有有效的方式以便互相进行监控，保护系统尽可能地避免遭受到 重大故障的损失。</p>
    <p>4、易管理性：不管是通过软件还是硬件方式的均衡解决方案，我们都希望它 有灵活、直观和安全的管理方式，这样便于安装、配置、维护和监控，提高工作 效率，避免差错。</p>
    <p>在硬件负载均衡设备上，目前主要有三种管理方式可供选择：（l)命令行接口 (CLI: Command Line Interface ),可通过超级客户端连接负载均衡设备串行接 口来管理，也能telnet远程登录管理，在初始化配置时，往往要用到前者；（2)图 形用户接口 （GUI: Graphical User Interfaces ),有基于普通web页的管理，也有 通过Java Applet进行安全管理， 一般都需要管理端安装有某个版本的浏览器； (3)SNMP ( Simple Network Management Protocol,简单网络管理协议）支持，通过 第三方网络管理软件对符合SNMP标准的设备进行管理。我们采用了第三种方式 进行管理。</p>
    <p>这些技术使我们实现了真正的业务无关性，并且能够实现所见即所得解决方 式，并且实现大用户量访问，多数据多操作同时访问。</p>
    <p>以上所述仅为本发明的较佳实施例，对本发明而言仅仅是说明性的，而非限 制性的。本专业技术人员理解，在本发明权利要求所限定的精神和范围内可对其 进行许多改变，修改，甚至等效，但都将落入本发明的保护范围内。</p>
