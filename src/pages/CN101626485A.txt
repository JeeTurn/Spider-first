    <p>基于中间件的IPTV扩展业务平台体系结构系统 &#32114;&#32177;</p>
    <p>本发明涉及网际协议电视（IPTV)技术，尤其涉及IPTV扩展业 务平台的构建技术。 背景技术</p>
    <p>IPTV业务系统在体系结构上分为3层：承载层、业务层、应用</p>
    <p>层。其中应用层、业务层主要由IPTV平台来实现，它包括内容运营 平台（COP)和业务运营平台（SOP)。在IPTV合作运营模式下，内 容运营平台是由内容运营商（CP)负责建设、维护和管理的平台， 包含EPG制作及管理系统、内容制作及发布系统、内容运营管理系 统、版权管理系统等功能模块，主要承担电子节目单管理、内容的管 理、用户的管理和视听节目计费管理功能。业务运营平台是由业务运 营商（SP)负责建设、维护和管理的平台，包含EPG服务系统、内 容分发系统（或称媒体交付系统）、运维支撑系统、门户导航系统、 安全管理系统、业务管理平台、扩展业务平台等功能模块，主要承担 将COP发布的视听节目和EPG节目单传送到用户终端、IPTV用户 的生命周期管理、IPTV业务的用户计费和收费支持、非视听节目业 务整合定价与包装、对视听与非视听节目的产品打包和销售等功能。 COP与SOP之间的接口遵循标准定义。</p>
    <p>IPTV的业务形式分为视频播放类基本业务和增值应用类扩展业务。目前IPTV的扩展业务已经有很多，例如家庭音视频点播、家 庭视频聚会、家庭多媒体分享、家庭网络支付、家庭教育、家庭医疗、 家庭导购、家庭炒股、家庭理财咨询、家庭保险咨询、社区游戏加交 友等。扩展业务平台为CP/SP提供了对扩展业务的处理、控制、存储、 发布以及对用户的集中呈现门户，属于业务运营平台的主要组成部 分。基于IPTV扩展业务平台将可以实现种种已有的或目前还没有的 增值业务。目前，这些扩展业务的部署大多数都是基于IPTV终端来 部署的，需要下载客户端软件，所以十分不利于快速部署、快速升级， 具有较大的局限性。而且由于各种增值业务实现的差异性极大，导致 目前不同厂商的IPTV终端与业务系统之间互联互通、增值业务的开 放式开发和快速部署的问题显得特别突出。</p>
    <p>为了解决这些问题，目前学术界和产业界的基本观点是采用中间 件技术。IPlV中间件架构的作用不仅是将应用程序和底层系统资源 隔离，更重要的是实现了应用与应用之间的交互。在IPTV系统中， 中间件完成了 IPTV终端（尤其是机顶盒）、IPTV平台系统、应用 三者之间的交互协作。</p>
    <p>IPTV中的中间件可以分为终端中间件和平台中间件两大部分。</p>
    <p>目前终端（尤其是机顶盒）中间件在国内外已经得到比较深入的 研究，例如欧洲DVB组织提出了 MHP中间件体系架构，ORCA提 出了 RiGHTv中间件体系架构，CCSA在机顶盒行业标准中定义了机 顶盒中间件结构，并有研究者探讨了基于IPTV机顶盒中间件实现的 客户端方式的增值业务】。机顶盒中间件技术为加快机顶盒上市、实现不同厂商的IPTV终端与业务系统之间互联互通提供了一条路径， 但是并没有解决好种类繁多的增值业务的开放式开发和快速部署的 问题，仍然缺乏灵活性。同时，IPTV终端厂商数量和已售出的IPTV 终端数量太多也给问题的解决带来了相当大的难度。</p>
    <p>另一种方案是采用平台中间件来解决，然而，由于扩展业务的差 异性、复杂性，每个业务可能都是由专业的软件提供商采用不同的技 术来实现，这使得定义并开发一套完整的平台中间件从技术上讲就已 经不可能了。</p>
    <p>WebOS (Web based operating system,网页操作系统），最早由 Syracuse大学NPAC的G C Fox等人于1995年提出，当时称之为 WebWindows。他们认为Web Windows将成为一个开放的、公用的和 模块化的Internet分布式操作系统。 一般认为，WebOS是一种基于浏 览器的虚拟的操作系统，用户通过浏览器可以在这个WebOS上进行 Web应用程序的操作。它包含以下几个部分：全局资源定位、广域文 件系统、安全远程进程控制（远程运行经过认证的进程，例如在客户 端通过浏览器执行服务器程序）、认证与安全。WebOS提供了在线 应用软件的执行环境，提供了用户管理网络计算机的管理环境。 WebOS将应用从复杂的分布式管理、网络编程中解脱出来，并为应 用提供了WebAPI，本质上就是一个中间件。由于WebOS往往集成 了一些门户、音视频在线播放、文件网络存储与分享、文件在线编辑、 网页游戏等服务，所以它可以被视为一种适合于部署、集成RIA(Rich Internet application)应用的集成中间件。说 发明内容</p>
    <p>本发明的目的提出了一种适于部署网页式增值业务的基于</p>
    <p>WebOS (Web based operating system)集成中间件系统的IPTV扩展 业务平台体系结构，是以服务端RIA方式为主来开展IPTV增值业务， 部署升级方便且对IPTV终端要求低，兼容性强。</p>
    <p>本发明基于中间件的IPTV扩展业务平台体系架构系统，包括下 层的资源层、中间层的WebOS集成中间件层和上层的应用层，所述 资源层包括操作系统、硬件和承载网络，用于完成硬件控制和网络传 输；所述WebOS集成中间件层完成底层资源透明化和合作对象的透 明化；所述应用层实现各种IPTV扩展应用。</p>
    <p>具体的，所述WebOS集成中间件层可以划分为WebOS业务执 行环境层和WebOS业务集成环境层两个子层。</p>
    <p>具体的，所述WebOS业务执行环境层用于扩展业务平台与业务 运营平台的其他功能模块的互通。</p>
    <p>具体的，所述WebOS业务执行环境层用于对CP/SP提供的第三 方RIA应用的集成。</p>
    <p>具体的，所述WebOS业务执行环境层提供WebOS内嵌应用引</p>
    <p>具体的，所述的WebOS业务集成环境层包括Web Services框架、 应用注册、应用发布、应用发现、应用生命周期管理、应用调度与协 同、应用间通信、网络文件系统等功能。具体的，所述的WebOS业务集成环境层包括一个Parlay X Web Services逻辑组件以便提供对传统电信业务的支持。</p>
    <p>本发明的基于中间件的IPTV扩展业务平台体系架构系统，以服 务端RIA方式为主来开展IPTV增值业务，部署升级方便且对IPTV 终端要求低。 附图说明</p>
    <p>图l是本发明具体实施方式基于中间件的IPTV扩展业务平台体 系架构系统的结构图。</p>
    <p>图2本发明基于新型扩展业务平台实现的增值业务的工作流程。 具体实施方式</p>
    <p>本发明基于WebOS集成中间件系统的IPTV扩展业务平台有以 下三个功能层。</p>
    <p>(1)资源层</p>
    <p>此层由操作系统、硬件、承载网络等软硬件资源构成，主要完成 硬件控制、网络传输等功能。</p>
    <p>(2 ) WebOS集成中间件层 WebOS集成中间件层主要完成底层资源透明化和合作对象 (IPTV终端上的客户端软件、SOP的其他功能模块、CP/SP应用） 透明化的功能。</p>
    <p>本层可以划分为WebOS业务执行环境（WebOS service execution environment, WSEE) 、 WebOS业务集成环境（WebOS service integration environment, WSIE)两个子层。WSEE子层主要完成Web Services框架、应用注册、应用发布、 应用发现、应用生命周期管理、应用调度与协同、应用间通信、网络 文件系统等功能。它提供的API称为WSEE API。 WSEE可以包含一 个可选的Parlay X Web Services逻辑组件以便提供对传统电信业务的 支持，这个组件可以部署在物理分离的网关上。Parlay X Web Services 是目前涉及电信服务的应用场景中最常用的应用程序接口 ，基于Web Services技术，主要完成需要电信服务提供商核心网络提供的以下通 信类服务：第三方呼叫、呼叫通知、短消息、多媒体消息、支付、账 户管理、终端状态、终端位置、呼叫处理、音频呼叫、多媒体会议、 地址列表管理和呈现。</p>
    <p>WSIE子层主要完成以下功能：</p>
    <p>第一是扩展业务平台与SOP的其他功能模块，如业务管理平台、 门户导航系统、媒体交互系统、运维支撑系统、安全管理系统等的互 通，并与这些功能模块协同工作。例如通过业务管理接口组件与业务 管理平台的互通，得到用户的开机认证服务授权结果、业务认证/授 权结果或得到COP的服务计费策略等；通过扩展门户或频道门户组 件与门户导航系统的互通为用户从导航门户重定向进入扩展业务平 台开放通道；通过媒体交互接口组件与媒体交互系统的互通为用户提 供媒体点播服务；通过运维支撑接口组件与运维支撑系统的互通达成 对IPTV终端的管理；通过安全管理接口组件与安全管理系统的互通 完成对用户使用业务时涉及安全/版权的管理等。这些互通的接口之 间都建议使用HTTP协议进行传输，并用SOAP封装远程过程调用RPC和输入输出参数，这样SOP的所有子系统都可以分布式部署和 处理]。</p>
    <p>第二是对CP/SP提供的第三方RIA应用的集成。它提供了自己 的客户端API、基于AJAX的客户端框架和类库、前台GUI (图形用 户接口）子系统、集成开发环境以便第三方CP/SP进行开放式开发， 提供了应用集成API以便第三方RIA应用能方便地集成到扩展业务 平台上来。为了安全性和部署便利性，CP/SP可以直接在自己的应用 服务器上部署RIA应用。WS正的扩展业务接入网关组件提供对 CP/SP的接入鉴权控制、资源分配和质量监控的功能。WSIE子层提 供的API统称为WSIE API。 ，</p>
    <p>第三是提供了 WebOS内嵌应用引擎（例如下载服务引擎、内容 存储与分享服务引擎、文档搜索服务引擎、文档在线编辑服务引擎、 多媒体音视频文件在线播放服务引擎等），某些引擎还可以Web服 务的方式提供一些SOAP方法给第三方开发者使用。 (3)应用层</p>
    <p>此层主要实现各种IPTV扩展应用或电信运营商的传统电信业</p>
    <p>务，可以由第三方提供。</p>
    <p>基于本发明扩展业务平台实现的增值业务的工作流程如下： 以家庭多媒体分享为例，IPTV扩展业务平台与IPTV终端（如</p>
    <p>机顶盒）的工作流程如图2所示，步骤如下。</p>
    <p>(1) 机顶盒开机向业务管理平台发起开机认证请求；</p>
    <p>(2) 业务管理平台向机顶盒返回认证鉴权结果；(3) 机顶盒向门户导航系统发出门户导航页面请求；</p>
    <p>(4) 门户导航系统向机顶盒返回门户导航页面内容；</p>
    <p>(5) 机顶盒通过门户导航系统向扩展业务平台发起访问请求 (省略从门户导航系统到扩展业务平台的重定向过程）；</p>
    <p>(6) 扩展业务平台向机顶盒返回扩展业务平台门户界面；</p>
    <p>(7) 机顶盒在扩展业务平台门户界面上选择多媒体文件分享业 务，向扩展业务平台发起多媒体文件分享业务选择请求；</p>
    <p>(8) 扩展业务平台向业务管理平台请求对用户进行业务鉴权和</p>
    <p>(9) 业务管理平台向扩展业务平台返回确认信息和用户信息；</p>
    <p>(10) 扩展业务平台向多媒体文件分享业务发起业务使用请 求，可以有以下两种方式。</p>
    <p>&#21443;方式（a):如果多媒体文件分享业务是采用WebOS集成 中间件开发环境开发的紧耦合应用（功能更多、交互效果 更好），则通过WSIE API调用发起业务使用请求，转步 骤（11)。</p>
    <p>*方式（b):如果多媒体文件分享业务是松耦合的普通Web 应用，可以简单地通过HTTP重定向发起业务使用请求， 转步骤（13)。</p>
    <p>(11) 多媒体文件分享业务接受扩展业务平台发起的请求并 将响应界面信息（如URL)通知扩展业务平台；</p>
    <p>(12) 扩展业务平台将多媒体文件分享业务的响应界面传送给机顶盒浏览器；</p>
    <p>(13 )      用户开始使用多媒体文件分享业务。</p>
