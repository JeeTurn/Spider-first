    <p>一种基于J2EE架构的监控系统和方法</p>
    <p>技术领域</p>
    <p>[0001]	本发明涉及一种基于 J2EE (Java 2 Platform Enterprise Edition, Java 2 平台企业版）架构的监控系统和方法，可以实现对操作系统、JVM(Java Virtual Machine, Java 虚拟机)、WEB容器及其他组件的状态进行监控。</p>
    <p>背景技术</p>
    <p>[0002]	随着互联网技术的飞速发展，在各个领域中开发的应用程序的规模不断的扩大， 开发周期、维护成本、性能的要求等都成为必须重点考虑的因素。特别是在电信、医疗等行 业中大型应用占据着主导地位。在JAVA开发的程序运行过程中，对操作系统，JVM, TOB容 器，数据库都是监控的主要对象。在日常维护中利用监控系统，可以对系统性能进行调优， 快速定位故障，对整个环境运行状态进行集中监控，而利用JAVA的可以跨操作系统平台的 特性，能够更好的开发应用服务。</p>
    <p>[0003]	目前的监控主要采用以下三种方式：</p>
    <p>[0004]	一是通过业务写日志的方式。</p>
    <p>[0005]	二是通过JMX (Java Management Extensions，Java管理扩展）的方式来直接交互。</p>
    <p>[0006]	三是通过使用专用的三方性能监控软件如JProfile等软件。</p>
    <p>[0007]	其中通过日志来监控只能监控到业务本身，只能用来判断系统运行的状态，而并 没有考虑到其他部分对业务的影响。JMX直接交互则可以通过接口来全面的监控，即在运 行过程中部署大量的组件，但是这样却降低系统的灵活性、可扩张性，加大了系统运行的负 担，降低了系统运行的性能。</p>
    <p>[0008]	使用JProfile工具进行系统运行监控需要将JProfile部署于系统运行环境当 中。这样的工具本身就需要占用大量的系统资源，只能是在需要时才运行。如果部署到真 实的运行环境中必然会导致服务器负载过重，降低了系统运行的性能。</p>
    <p>发明内容</p>
    <p>[0009]	本发明所要解决的技术问题是提供一种基于J2EE架构的监控系统和方法，监控 WEB容器、JVM、操作系统、业务资源等组件的数据信息，对整个系统环境不同层次进行数据 采集、监视、诊断。将监控对应用系统的性能影响降到最低限度，增强监控系统的灵活性、可 扩张性。</p>
    <p>[0010]	为解决上述技术问题，本发明提供一种基于J2EE架构的监控系统，包括：设置在 被监控设备上的监控代理端和设置在J2EE WEB容器上的监控服务端；所述监控代理端上 设置至少一个包括插件和对应的插件配置文件，通过所述插件获取被监控设备的相关数 据，所述监控代理端与所述监控服务端之间可交互控制指令和数据；所述监控服务端用于 对监控代理端进行注册管理，以及对接收到的被监控设备的相关数据进行分析处理。</p>
    <p>[0011]	进一步的，所述监控代理端包括：管理应用模块，用于与所述监控服务端之间进行控制指令和数据的交互，以及动态管理插件和与插件进行信息交互；插件应用模块，所述用于通过接口采集被监控设备的相关数据；配置文件模块，用于存放所述插件中，描述各插件 接口对应的配置文件。</p>
    <p>[0012]	进一步的，所述的监控服务端包括：服务应用模块，用于与所述管理应用模块交互 信息，以及对所述监控代理端进行注册管理，对相关数据进行分析处理；数据库模块，用于 存储配置信息、所述监控代理端的基本信息、监控数据以及分析处理后的数据。</p>
    <p>[0013]	优选的，所述监控系统还包括：与所述监控服务端相连的WEB服务器，用于以TOB 方式发出对被监控设备的控制信息，以及显示被监控设备的相关监控数据。</p>
    <p>[0014]	优选的，所述监控代理端上的插件通过以下方式中的至少一种获取被监控设备 的相关数据：JMX、JNI (Java Native Interface, Java 本地接 口）、RMI (Remote Method Invocation,远程方法调用)。</p>
    <p>[0015]	优选的，所述监控代理端上设有Socket端口，用于监听所述监控服务端发来的指 令。</p>
    <p>[0016]	本发明还提供了一种基于J2EE架构的监控方法，包括以下处理步骤：在被监控设 备上配置监控代理端，在J2EE WEB容器上配置监控服务端；在监控服务端上进行监控基本 信息配置，并启动监控服务端；监控代理端向所述监控服务端进行注册，并根据监控服务端 的指令和数据加载或配置插件和对应的插件配置文件，并向所述监控服务端反馈监控代理 端的基本信息；所述监控代理端通过所述插件获取被监控设备的相关数据，并与所述监控 服务端交互指令和数据完成对被监控设备的监控。</p>
    <p>[0017]	优选的，在监控处理过程中还包括以下处理步骤：TOB服务器以TOB方式发出对被 监控设备的控制信息，以及显示被监控设备的相关监控数据。</p>
    <p>[0018]	优选的，所述监控代理端通过所述插件获取被监控设备的相关数据具体为：通过 以下方式中的至少一种获取被监控设备的相关数据：JMX、JNI、RMI。</p>
    <p>[0019]	优选的，所述监控代理端通过其上设置的Socket端口监听所述监控服务端发来 的指令和数据。</p>
    <p>[0020]	本发明所述方法和系统，与现有技术相比，通过插件的方式大大降低了监控系统 运行时对系统的性能的影响，很灵活的监控到WEB容器、JVM、操作系统、业务资源等组件不 同层次的信息；此外还可以实现跨平台运行，集中管理，帮助开发人员快速定位故障、解决 问题，帮助运维人员通过监控手段提高系统运行的稳定性，加快开发时间和降低后期维护 成本。</p>

    <p>具体实施方式</p>
    <p>[0024]	本发明监控系统和方法是通过部署插件和配置文件结合的方式来实现的，下面通 过具体实施方式结合附图对本发明作进一步详细说明。[0025]	实施例一：</p>
    <p>[0026]	请参考图1，本实施例中基于J2EE架构的监控系统包括：设置在被监控设备上的监控代理端和设置在J2EE WEB容器上的监控服务端；图示中有两个监控代理端。在监控代 理端上设置至少一个插件和对应的插件配置文件，通过插件获取被监控设备的相关数据。 监控代理端上的插件可以通过以下方式中的至少一种获取被监控设备的相关数据：JMX、 JNI、RMI。监控代理端与监控服务端之间可交互控制指令和数据，例如监控代理端上可以通 过启动Socket端口，来监听所述监控服务端发来的指令和数据。</p>
    <p>[0027]	监控服务端用于对监控代理端进行注册管理，以及对接收到的被监控设备的相关 数据进行分析处理。</p>
    <p>[0028]	图示中监控代理端包括以下主要模块：管理应用模块、插件应用模块和配置文件 模块。其中管理应用模块，用于与监控服务端之间进行控制指令和数据的交互，以及动态管 理插件和与插件进行信息交互。插件应用模块，由至少一个插件集合构成插件应用模块，用 于通过接口采集被监控设备的相关数据；配置文件模块，用于存放所述插件中，描述各插件 接口对应的配置文件。</p>
    <p>[0029]	图示中监控服务端包括以下主要模块：服务应用模块，用于与管理应用模块交互 信息，以及对监控代理端进行注册管理，对相关数据进行分析处理；数据库模块，用于存储 配置信息、监控代理端的基本信息、监控数据以及分析处理后的数据。</p>
    <p>[0030]	此外，还可以设置一个TOB服务器（图1中未示出），将其与所述监控服务端相连， 用于以WEB方式发出对被监控设备的控制信息，以及显示被监控设备的相关监控数据。</p>
    <p>[0031]	实施例二：</p>
    <p>[0032]	请参考图2，本实施例中基于J2EE架构的监控方法，包括以下处理过程：</p>
    <p>[0033]	步骤一，在被监控设备上配置监控代理端，在J2EE WEB容器上配置监控服务端；</p>
    <p>[0034]	步骤二，在监控服务端上进行监控基本信息配置，并启动监控服务端；</p>
    <p>[0035]	步骤三，监控代理端向所述监控服务端进行注册，并根据监控服务端的指令和数 据加载或配置插件和对应的插件配置文件，并向监控服务端反馈监控代理端的基本信息； 监控代理端向监控服务端发送信息可以通过HTTP协议进行传输，监控服务端向监控代理 端发送信息则可以通过TCP/IP协议进行传输；</p>
    <p>[0036]	步骤四，监控代理端通过所述插件获取被监控设备的相关数据，并与监控服务端 交互指令和数据完成对被监控设备的监控。</p>
    <p>[0037]	下面再对插件的实现过程进行详细说明，本发明的插件是将相同类型的接口打成 一个独立的jar包，没有任何依赖。如：要获取操作系统的信息和tomcat的信息，那就把获 取操作系统的接口和获取tomcat信息的接口打两个jar包，分别命名SyStem_plugin. jar 和 tomcat_plugin. jar。</p>
    <p>[0038]	实施例二 ：</p>
    <p>[0039]	请参考图3，本实施例中基于J2EE架构的监控系统结构，图示中监控代理端（以下 简称Agent)和监控服务端（以下简称Server)分布在两台服务器上，Agent和Server也 可以分布在一台物理机器上。</p>
    <p>[0040]	图示中0001和0014分别代表Agent和Server的操作系统。0002和0015分别代 表Agent和Server的JVM。0003是Agent管理应用模块，部署在JVM下，主用来跟Server交互和运行时检测插件及获取插件数据。0004是部署在JVM下的Java应用程序，被监控对象，和Agent部署在同一台机器，独立于Agent。0005是非Java应用程序，被监控对象， 和Agent部署在同一台机器，独立于Agent。0006是用来获取0004应用数据的监控插件， 当Server发起请求命令，Agent接收到后调用插件接口来获取数据返回给Server。0007， 0008,0009分别对应的是JVM，非Java应用，操作系统的插件，原理和0006 &#8212;样。0010，0011 是调用方式，当Web插件调用Web用接口的时候是通过JMX的方式来调用，两个模块是低耦 合。以上0006，0007，0008，0009是Agent端的主要核心，这些插件只是举例，还可以添加很 多项目的插件。Agent运行的时候可以删除任何一个插件或多个，也可以添加别的插件都不 会引起Agent的正常运行，Agent启动后会自动启动插件探测器检测插件是否存在。0012 是Server到Agent的传输协议，本实施例中采用的TCP/IP协议。0013是Agent到Server 的传输协议，本系统采用的HTTP协议。0016是J2EE Web容器，基于JVM下，本实施例采用 的是开源的TOMCAT。0017为Server端工作模块，部署J2EE容器下。Server发出请求命 令后，接收到Agent发送过来的监控数据，将数据进行保存，分析，诊断。0018是一个浏览 器或有相同功能的装置，主要用来管理Server端，监控指标，查看监控数据，分析及诊断结 果。0019为数据库，部署在Server，也可以独立部署在另外的服务器。</p>
    <p>[0041]	本实施例的监控处理步骤如下：</p>
    <p>[0042]	步骤1 ：启动Server端0017，Server工作模块读取数据库相关信息，检测是否有 Agent存在。</p>
    <p>[0043]	步骤2 ：启动Agent管理应用模块0003，注册到Server上，Agent启动插件探测器 检测插件0006，0007，0008，0009是否存在，Agent将检测到的数据及Agent的基本信息上 传输到Server端保存。</p>
    <p>[0044]	步骤3 ：到Server端登陆浏览器0018，查看每台Agent发送过来检测到可以被监 控的信息指标。</p>
    <p>[0045]	步骤4 ：在浏览器0018上操作，添加将要被监控的监控指标或删除正在被监控的 指标。此时，0006，0007，0008，0009插件开始处于被监控状态。</p>
    <p>[0046]	步骤5 =Agent管理应用模块运行时检测Agent插件状态，并向Server发送最新的 插件状态。</p>
    <p>[0047]	步骤6 =Server定时或实时向Agent端发送被监控指标相对应的执行，Agent获取 到命令后将采集相应的监控指标返回到Server，Server端对监控数据保存，分析，诊断。</p>
    <p>[0048]	步骤7 =Server端获得数据显示在浏览器0018上。</p>
