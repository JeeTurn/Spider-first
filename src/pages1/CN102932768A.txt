    <div class="abstract">本发明实施例公开了一种用于有费率切换的呼叫接续过程中的处理方法和装置，其中，所述方法包括：接收到交换机发送的为用户鉴权请求，其中，所述用户鉴权请求是由主叫发起呼叫触发的；为用户鉴权；当鉴权通过后，发送监控呼叫状态的消息；接收到主被叫建立呼叫的消息；以接收到主被叫建立呼叫的消息的时间作为开始计费的时间点，进行计费预算；根据计费预算，对交换机进行后续控制。本发明实施例中智能网设备主动监控被叫应答消息，智能网设备接收到主被叫建立呼叫的消息，以智能网设备接收到主被叫建立呼叫的消息的时间作为开始计费的时间点，使得智能网设备能够准确的获知呼叫开始的时间，该方案能够准确的对呼叫进行计费。</div>

    <p>一种呼叫接续过程中的处理方法和装置</p>
    <p>技术领域</p>
    <p>[0001]	本发明涉及通信技术领域，具体涉及一种用于有费率切换的呼叫接续过程中的处理方法和装置。</p>
    <p>背景技术</p>
    <p>[0002]	准确地实现对用户呼叫计费是运营商健康运营的基本，尤其是在呼叫接续过程中存在费率切换的情况，是对运营商侧计费系统的重大考验。在对用户呼叫过程中的计费操作是由核心网设备与智能网设备共同完成的。其中，核心网中的主叫所在的移动交换中心业或业务交换点（MSC/SSP, MobileSwitching Center/Service Switching Point)将呼叫接续的开始时间通过启动检出点（DPI, Initial Detection Point)发送给智能网中业务控制点（SCP, Service Control Point),由SCP进行鉴权,预留用户金额和对授权的用户可用呼叫时长分片等处理，分配给主叫用户呼叫时长，且在余额用尽后通知主叫侧MSC/SSP结束主叫与被叫的通话。</p>
    <p>[0003]	在上述主叫侧MSC/SSP与SCP之间的通信通常是采用移动网络增强定制应用逻辑应用部分（CAP, CAMEL (Customized Application for Mobile Network Enhanced logic)Application Part)信令,现有技术中，对于首个授权呼叫时长分片，计费开始时间是以启动检测点消息（IDP，Initial Detection Point)上报的呼叫开始时间为依据。主叫侧MSC/SSP按照SCP下发的授权的呼叫时长分片进行呼叫监控。</p>
    <p>[0004]	由于主叫侧MSC/SSP上报的IDP中的呼叫开始时间，并非主叫侧与被叫侧建立的&#183;连接的时刻，即真实的呼叫接通的时间，IDP中的呼叫开始时间早于双方接续通话，在该时间差内涉及振铃以及被叫摘机时延，但实际上，主叫用户的业务开始时间是从呼叫接续时间开始，因此，由于现有技术中，SCP在计费时不能依据业务实际的开始时间计费，导致计费的不准确。</p>
    <p>[0005]	尤其是，对于在首个呼叫时长分片中配置了呼叫费率切换时，即在首个呼叫时长分片中费率由便于到贵，(例如，首个呼叫时长分片为5分钟，恰好在分片的2分51秒时，该分片跨过了 12点，则前2分51秒按照每秒8分收费，而分片的后2分09秒按照每秒15分收费。）由于SCP在计费时不能依据业务实际的开始时间计费，导致计费的不准确，使得运营商遭受损失。</p>
    <p>[0006]	例如：在SCP中的计费配置为在11:40:00前费率为9分/s，之后费率为17分/s，用户A余额为500分，在11:39:33秒的时候发起呼叫，此时授权为27s+15s=42s (27s为11:40:00之前的时间，15s为11:40:00之后的时间，该呼叫使用金额498分);用户B在11:39:42的时候接听了电话，A和B呼叫了 42s ;根据呼叫开始时间11:39:42进行计费，前18s按9分/s扣费，后24s按17分/s扣费，就会发现余额不足，只能扣37s，所以话单的呼叫持续时间（call duration)为42s,付费时长（charging duration)为37s。用户通话了42s，而运营商仅能收到37s的钱,导致运营商损失。发明内容</p>
    <p>[0007]	本发明实施例提供了一种呼叫接续过程中的处理方法和装置，SCP在计费时依据业务实际的开始时间计费，克服了在有费率切换时计费的不准确的问题。</p>

    <p>具体实施方式</p>
    <p>[0044]	实施例一</p>
    <p>[0045]	本发明实施例提供了一种呼叫接续过程中的处理方法，不仅使用全球移动通讯系统（GSM, Global System of Mobile communication)网络，该思想也可以运用在宽带码分多址（WCDMA, Wideband Code Division Multiple Access),或者是码分多址（CDMA, CodeDivision Multiple Access)网络的智能呼叫计费。由于对于不同的网络，会有不同的信令，本方案的核心思想是在智能网设备监控到交换机已经建立了呼叫连接后，以交换机反馈的监控到呼叫连接建立的时间，作为计费开始时间，从而保证了计费开始时间的准确，对于有费率切换的情况，准确的计费开始时间使得计费更加准确，避免了运营商的损失。</p>
    <p>[0046]	如图I所示，该方法包括：</p>
    <p>[0047]	步骤101 :接收到交换机发送的为用户鉴权请求，其中，该请求是由发起呼叫触发的；</p>
    <p>[0048]	步骤102 :为用户鉴权；</p>
    <p>[0049]	步骤103 :当鉴权通过后，发送监控呼叫状态的消息；</p>
    <p>[0050]	其中，不同的网络，具体监控的呼叫状态名称不同，具体信息也可以不同。</p>
    <p>[0051]	步骤104 :接收到主被叫建立呼叫的消息；[0052]	其中，由于智能网设备能够对呼叫进行监控，当前呼叫建立连接后，交换机主动发送消息通知智能网设备，使得智能网设备可以以更为接近呼叫真是建立连接的时间点，作为计费的开始时间，当在呼叫过程中有费率切换的时候，对费率前后的计费时间更为准确。以GSM网络为例，所述智能网设备接收到主被叫建立呼叫的消息可以理解理解为：SCP接收交换机发送的包括有被叫应答消息的事件报告基本呼叫状态模型（ERB，Event ReportBCSM (Basic Call State Model));以CDMA网络为例，述智能网设备接收到主被叫建立呼叫的消息可以理解理解为：SCP接收到交互机发送的包括有被叫应答消息的O响应（SPOANSffER)ο</p>
    <p>[0053]	步骤105 :以接收到主被叫建立呼叫的消息的时间作为开始计费的时间点，进行</p>
    <p>计费预算；</p>
    <p>[0054]	对于不同的网络技术，智能网设备的计费预算会有很大区别，当前仅举例GSM网络和CDMA网络。若GSM网络，智能网设备会将呼叫划分为多个可用分片时间来控制呼叫；若CDMA网络，智能网设备在呼叫开始时将用户可用的最大时间长发送给交换机，在智能网设备进行监控，具体可以是智能网设备接收到所述交互机发送的包括有被叫应答消息的O响应，其中，所述包括有被叫应答消息的O响应用于表示主被叫建立呼叫。此次仅是便于理解本发明的举例，并非穷举。</p>
    <p>[0055]	步骤106 :根据计费预算，对交换机进行后续控制。</p>
    <p>[0056]	需要理解的是，对于不同的网络环境，智能网设备对交换机进行后续控制的技术有所不同。</p>
    <p>[0057]	通过上述对本发明实施例提供的一种呼叫接续过程中的处理方法的说明，该方法中智能网设备主动监控被叫应答消息，智能网设备接收到主被叫建立呼叫的消息，以智能网设备接收到主被叫建立呼叫的消息的时间作为开始计费的时间点，使得智能网设备能够准确的获知呼叫开始的时间，该方案能够准确的对呼叫进行计费。</p>
    <p>[0058]	实施例二</p>
    <p>[0059]	本发明实施例提供了一种呼叫接续过程中的处理方法，如图2所示，该方法包括：</p>
    <p>[0060]	步骤201 =SCP接收交换机发送的启动检测点消息；</p>
    <p>[0061]	其中，该启动检测点消息可以是在主叫侧交换机接收到主叫用户的呼叫请求后，触发发送启动检测点消息给SCP，从而触发智能网中SCP设备对呼叫的控制，该控制包括计费控制，也可以包括呼叫控制等。</p>
    <p>[0062]	步骤202 =SCP根据IDP中用户信息，对用户进行鉴权；</p>
    <p>[0063]	步骤203 :当鉴权通过后，发送请求报告基本呼叫状态模型事件（RRBE，RequestReport BCSM (Basic Call State Model) Event)给交换机；所述RRBE中至少包括监控被叫应答消息；</p>
    <p>[0064]	需要理解的是，SCP发送包括有监控被叫应答消息的RRBE，当主叫侧交换机与被叫侧交换机建立呼叫连接是，及时上报该RRBE，使得SCP可以准确的获知呼叫的起始时间，便于计费；尤其针对在SCP为呼叫分配的第一个可用分片时间中有费率切换的情况，SCP能够准确知道在分配的第一个分片时间中费率切换前的通话时长，以及费率切换后的通话时长，避免由于计费不准确导致的运营商损失。</p>
    <p>[0065]	步骤204 =SCP接收交换机发送的包括有被叫应答消息的事件报告基本呼叫状态模型（ERB, Event Report BCSM (Basic Call State Model))；</p>
    <p>[0066]	步骤205 ：SCP以接收到的包括有被叫应答消息的ERB的时间作为开始计费的时间点，进行计费预算，其中，包括有被叫应答消息的ERB用于表示主被叫建立呼叫；</p>
    <p>[0067]	步骤206 :SCP根据计费预算,发送第一个申请计 费（AC, Apply Charging)给交换机；</p>
    <p>[0068]	步骤207 ：SCP根据交换机发送的信息进行后续控制。</p>
    <p>[0069]	其中，步骤207中后续控制包括对呼叫计费的控制，例如：当第一个AC中的可用分片时间用尽，SCP下发第二个AC为用户提供可用分片时间继续呼叫，也可以包括对呼叫链路的控制，例如：当SCP从交换机处接收到呼叫断开的信息，控制交换机释放呼叫连接。</p>
    <p>[0070]	还需要理解的是，步骤207中的后续控制与现有技术相同，此处不多述。</p>
    <p>[0071]	通过上述对本发明实施例提供的一种呼叫接续过程中的处理方法的说明，该方法中SCP主动监控被叫应答消息，即发送包括有监控被叫应答消息的RRBE，当接收到表征主被叫交换机建立呼叫的响应（即ERB)后，以接收到ERB时间作为开始计费的时间点，使得SCP能够准确的获知呼叫开始的时间，对于第一个可用分片时间中有费率切换的情况，该方案能够准确的对呼叫进行计费。</p>
    <p>[0072]	进一步，步骤203中RRBE还可以包括：其它信息等，使得SCP可以更准确的监控呼口q，并作出合理的控制。</p>
    <p>[0073]	进一步，在步骤203之后，步骤204之前，所述方法还包括：</p>
    <p>[0074]	步骤208 :当SCP根据第一个AC中可用分片时间和预置的费率切换时刻，判断在第一 AC中可用分片时间中是否有费率切换；如果有，则执行步骤204 ;如果没有，则执行步骤 209 ：</p>
    <p>[0075]	步骤209 =SCP以接收到IDP时间作为开始计费的时间点，进行计费预算；根据计费预算，发送第一个AC给交换机；根据交换机发送的信息进行后续控制。</p>
