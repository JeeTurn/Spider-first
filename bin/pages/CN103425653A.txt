    <div class="abstract">本发明公开一种实现DICOM影像二次检索的方法及系统。其中，所述方法包括步骤：对DICOM影像进行初次检索，将初次检索结果保存在Web客户端的缓存；在Web客户端输入二次检索条件，以初次检索结果为检索对象进行二次检索；由Web客户端将二次检索结果对应的影像信息组合成WADO协议串并传送给WADO服务，通过WADO服务从Web&#160;DICOM服务器提取对应的DICOM影像并显示DICOM影像。本发明采用初次检索数据，减少与数据库服务器交互次数，缩小了检索范围，具有检索效率高、检索速度快且检索方便的特点，是一种有效的医学影像检索方式。</div>

  </div></div><div class="patent-section patent-description-section"><div class="patent-section-header"><span class="patent-section-title"> 说明</span></div><div class="patent-text"><div mxw-id="PDES63997115" lang="ZH" load-source="patent-office" class="description">
    <p>实现DICOM影像二次检索的方法及系统</p>
    <p>技术领域</p>
    <p>[0001]	本发明涉及一种DICOM影像检索技术，尤其是涉及一种基于WADO协议实现DICOM影像快速二次检索的方法及系统。</p>
    <p>背景技术</p>
    <p>[0002]	由于医用医学影像数量庞大，面对海量的数字影像，如何能够从中快速有效的找到有用的影像信息，成为了浏览医学影像的难题。基于此而出现了影像检索技术。</p>
    <p>[0003]	传统的方法，通过人工对影像进行文字标识，再用关键字来检索影像。这种方法简单快捷，但是影像的形状，纹理，空间关系等很难用文字描述。而且人工标识，效率低下，带有主观偏差。随着技术的发展，90年度出现了基于影像特征的检索技术。但该技术虽然在能够较好的识别影像，但是其计算量庞大，计算效率不高。</p>
    <p>[0004]	现在的检索技术大部分都是基于DICOM影像的文本信息与影像信息结合的检索。但是检索中每次都跟数据库进行交互，浪费了大量的时间。另外这种检索方式不能够进行模糊检索查询，要么输入准确的信息，要么就填写为空。如果只是记住了某个人的名字忘记了姓氏，就很难实现检索。</p>
    <p>发明内容</p>
    <p>[0005]为了解决现有基于DICOM影像的文本信息与影像信息结合的检索时存在检索不精确且检索效率低的缺陷，本发明提出一种基于WADO协议的DICOM影像二次检索方法，二次查找的过程不需要在于数据库进行交互，节省了检索时间，降低了服务器的负荷，提高检索效率。</p>
    <p>[0006]	本发明采用如下技术方案实现:一种实现DICOM影像二次检索的方法，其包括步骤:</p>
    <p>[0007]	对DICOM影像进行初次检索，将初次检索结果保存在Web客户端的缓存；</p>
    <p>[0008]	在Web客户端输入二次检索条件，以初次检索结果为检索对象进行二次检索；</p>
    <p>[0009]	由Web客户端将二次检索结果对应的影像信息组合成WADO协议串并传送给WADO服务，通过WADO服务从Web DICOM服务器提取对应的DICOM影像并显示DICOM影像。</p>
    <p>[0010]	其中，所述对DICOM影像进行初次检索的步骤包括:</p>
    <p>[0011 ] 将DICOM文件解析成DICOM文本信息和DICOM影像，DICOM文本信息存储在Web服务器的数据库，而DICOM影像存储在Web服务器的影像文件夹；</p>
    <p>[0012]	在Web客户端输入初次检索条件，通过与Web DICOM服务器交互，由Web DICOM服务器根据初次检索条件在数据库中检索DICOM文本信息；</p>
    <p>[0013]	由Web DICOM服务器向Web客户端返回符合初次检索条件的初次检索结果。</p>
    <p>[0014]	其中，所述初次检索结果以列表方式进行显示。</p>
    <p>[0015]	其中，所述二次检索条件为姓名，对姓名检索提供汉字、拼音、与英文的混合检索。</p>
    <p>[0016]	其中，以姓名为二次检索条件进行二次检索的步骤包括:[0017]	在Web客户端提供的检索框中输入姓名；</p>
    <p>[0018]	从初次检索结果中提取所有的姓名信息，将提取的姓名信息与检索姓名进行匹配检索；</p>
    <p>[0019]	获得与检索姓名相匹配的检索结果。</p>
    <p>[0020]	其中，所述二次检索条件为检查编号或病人编号，当输入检查编号或病人编号后，将检查编号或病人编号作为一个整体字符串与初次检索结果中的信息进行匹配检索。</p>
    <p>[0021]	其中，所述二次检索条件为性别或检查类型，当输入性别或检查类型后，与初始检索结果中的信息进行比较，将全匹配的信息显示为二次检索结果。</p>
    <p>[0022]	其中，所述二次检索条件为姓名、检查编号、病人编号、性别和检查类型至少其中之二的组合。</p>
    <p>[0023]	另外，本发明提出一种实现DICOM影像二次检索的系统，所述系统包括Web客户端与存储DICOM文件的Web DICOM服务器，且采用前述方法实现DICOM影像二次检索。</p>
    <p>[0024]	与现有技术相比，本发明具有如下有益效果:</p>
    <p>[0025]	本发明采用初次检索数据，减少与数据库服务器交互次数；另外，二次检索中可以进行模糊查找，不用输入详细内容，或者完全空白内容，缩小查找范围。因此，本发明提出的DICOM影像检索方法，检索效率高、检索速度快且检索方便，是一种有效的医学影像检索方式。</p>

    <p>具体实施方式</p>
    <p>[0037]	为了便于理解，以下先介绍DICOM文件的组成/WADO协议以及DICOM影像初次检索技术分别介绍。。</p>
    <p>[0038]	一、有关 DICOM 文件</p>
    <p>[0039]	DICOM文件一般由一个DICOM文件头和一个DICOM数据集合组成，如图1所示。DICOM文件头包含了标识数据集合的相关信息。文件头的最开始是文件前言，它由128组OOH字节组成，接下来是DICOM前缀，它是一个长度为4字节的字符串“DIC0M”，可以根据该值来判断一个文件是不是DICOM文件。文件头中还包其它一些非常有用的信息，如文件的传输格式、生成该文件的应用程序等。</p>
    <p>[0040]	DICOM数据集合由DICOM数据元素按照一定的顺序排列组成，它按照DICOM标准的PS3.5部分来编码，不仅包括医学图像，还包括许多和医学图像有关的信息，如病人姓名、图像大小等，其格式如图2所示。</p>
    <p>[0041]	数据元素是DICOM文件内容的最基本单元，它由四个部分组成:标签、数据描述(VR, Value Representation)、数据长度和数据域，其格式如图3所示。</p>
    <p>[0042]	DICOM文件包含了病人的各种详细信息(简称DICOM文本信息)以及图像信息(以下称为DICOM影像)，DICOM文本信息较小，可以存入数据库方便检索，但是DICOM影像的信息量非常大，将这些信息存入数据库是不现实的，我们根据Study Instance UID, SeriesInstance UID, SOP Instance UID等信息将DICOM文件分类存入本地文件夹当中。我们通过检索数据库中的信息调用文件夹的影像信息，实现对DICOM影像的信息检索。</p>
    <p>[0043]	二、有关WADO协议</p>
    <p>[0044]	WADO 是 Web Access to DICOM Persistent Objects 的简称。是 DICOM 协议的一部分。在WADO标准的规定中Web环境镇南关的通信实体间的属于描述如下:</p>
    <p>[0045]	DICOM持久性对象。他是一种分配了具体的SOP实例ID作为唯一标识的数据对象，它作为对象被安全的保存一段时期。在DICOM标准里面，DICOM持久性对象属于符合服务对象对(Service Object Pair, SOP)实例。</p>
    <p>[0046]	Web客户端。使用互联网技术通过HTTP/HTTPs协议对WebDICOM服务上的DICOM持久性对象实现访问的端系统。</p>
    <p>[0047]	DICOM Web服务器。同时管理DICOM持久性对象，并根据Web客户端请求传送对象的服务器系统。</p>
    <p>[0048]	WADO服务是Web客户端通过HTTP/HTTPS协议通信，从Web DICOM服务器找回DICOM持久性对象的过程。其中DICOM Web服务器从DICOM持久性对象进行归档管理的系统，并负责相应客户端的请求发送数据。WADO的个体请求是由请求地址(WAD0服务器)和请求参数组成。其具体传输语法如下:〈通信协议&gt;://服务器地址/路径？请求参数</p>
    <p>[0049]	三、有关DICOM影像初次检索技术</p>
    <p>[0050]	现在的DICOM影像检索是基于DICOM文本信息和DICOM影像信息相结合的来实现的。在影像信息检索之前，需要将DICOM文件解析成DICOM文本信息和DICOM影像信息。其中DICOM文本信息较小，同时为了便于检索，需要存入Web服务器的数据库当中，而DICOM影像信息庞大，需要根据DICOM文本信息，建立影像文件夹，按照Study Instance UID,Series Instance UID、SOP InstanceUID等信息对影像信息进行归档,方便检索。</p>
    <p>[0051]	如图4所示，本发明提出一种基于DICOM影像检索与本地信息检索结合的一种快速二次查找DICOM影像的技术。在图4所示的DICOM影像检索的设计框图中，粗线条表示初次检索的处理流程，细线代表二次检索的处理流程。Web客户端与数据库的交互次数直接决定了影像检索的速度。为了提高检索速度，我们将初次检索的信息存储到本地的缓存当中，在二次检索的时候，不用再与服务器数据库交互，直接使用初次检索的数据通过WADO服务查找影像，或者搜索信息。这样节省了 二次检索与数据库交互的时间，从而提高检索效率。</p>
    <p>[0052]	具体来说，DICOM影像初次检索即Web客户端与DICOM Web服务器(以下又简称服务器)的数据库之间的一次数据交互的过程。结合图5所示，在Web客户端输入初次检索条件(比如:性别、检查类型等等)，通过与服务器交互，让服务器根据客户端输入的条件在数据库中检索DICOM文本信息，向Web客户端返回符合条件的患者信息(病人姓名、性另U、检查类型、影像信息(包括StudyUID，SeriesUID, SOPInstanceUID)等)。这些信息以JSON(JavaScript Object Notation)数据结构返回到Web客户端,存入Web客户端的(本地)缓存，并将易于检索的信息显示在初次检索结果的列表List (List用来显示检索信息的变量)当中，初次检索结果的列表List中包含姓名、性别、检查编号、检查类型、检测日期，影像信息等等信息。在初次检索的信息当中，影像信息以一个三维数组的形式出现，在列表List中的结构形式如下:</p>
    <p>[0053]</p>
    <div class="patent-image small-patent-image"> <a href="//patentimages.storage.googleapis.com/CN103425653A/CN103425653AD00061.png"> <img id="idf0001" file="CN103425653AD00061.tif" img-content="drawing" img-format="tif" src="//patentimages.storage.googleapis.com/CN103425653A/CN103425653AD00061.png" class="patent-full-image" alt="Figure CN103425653AD00061"> </a> </div>
    <p>[0055]	这种结构既可以方便取出影像信息，又便于组成每个影像文件的WADO协议串。</p>
    <p>[0056]	DICOM影像初次检索得到的检索结果范围较广，内容较多，需要从这些初次检索结果中查找到有用的信息。如图6所示，DICOM影像的二次检索是基于在初次检索的基础上进行的再次检索，即在初次检索结果中作进一步的检索。可以在姓名、检查编号、病人编号、性别、检查类型等检索框中输入二次检索条件。这种方式可以有效缩小检索范围，同时由于此次检索是在Web客户端的本地缓存的检索，不需要与DICOM Web服务器的数据库再次进行交互，节省了与数据库交互的过程，检索效率大大提高。</p>
    <p>[0057]	二次检索当中包括以下几种方式:</p>
    <p>[0058]	1、姓名检索</p>
    <p>[0059]	姓名检索，属于汉字、拼音、与英文的混合检索。既可以输入汉字(如果姓名是汉字)汉字拼音，汉字拼音的首字母，或者英文的首字母等，都可以检索到符合条件的信息。</p>
    <p>[0060]	姓名中的信息最为复杂，其中包括了汉字、拼音、英文单词等等这些内容，如果用简单的匹配方式根本行不通。而且要考虑输入的字符串不管是汉字还是拼音或者拼音简写都可以将所需的信息检索出来。根据这些情况我们采用了汉字转拼音技术，将汉字转化为拼音。从拼音汇总提取出各种可能的简写组合。然后将输入的内容与各种组合比对，检索是否有相匹配的组合，有的话此信息则为匹配信息，否则将输入字符串与汉字匹配，如果还没有匹配的话则说明此信息不是查找的信息，进行下一条信息的检索，以此类推检索所有在列表中的信息。</p>
    <p>[0061]当检索框中输入姓名时，触发搜索框的change事件，在搜索框中首先将缓存中初次检索结果的姓名信息提取出来，然后进入匹配检索。具体流程如图7所示。其中的姓名是病人姓名；List是是初次检索结果列表；match是匹配检索的内容。</p>
    <p>[0062]	2、检查编号和病人编号检索</p>
    <p>[0063]	检查编号或病人编号检索属于同一类的字符串匹配检索。如果病人信息的编号中存在输入的字符串则显示该序列。具体流程如图8所示。其中的编号是病人编号者是检查编号；List是初次检索结果列表；match是匹配检索的内容。</p>
    <p>[0064]	检查编号话和病人编号检索，属于数字字符串检索。需要能够检索到包含所输入的数字字符串的信息。将输入的数字字符串与初次检索结果中的每条相应的信息进行对t匕，检索其中是否包含输入的信息串。注意:此处是将输入字符串作为一个整体，不是每个数字的匹配，比如0022可以匹配1002200但不能匹配001220。这种方式，只要使用者能够记得编号中的部分数字串就可以查找需要的信息。</p>
    <p>[0065]	3、性别和检查类型检索</p>
    <p>[0066]	性别或检查类型的检索属于完全匹配类型的检索。只有在输入的值完全匹配初次检索结果中的信息才能够显示出来。其具体流程如图9所示。其中性别、检查类型表示输入的检查类型；List是初次检索结果列表；match是匹配检索的内容。</p>
    <p>[0067]	检查类型和性别的检索属于全匹配类型的检索。按钮按下时将按钮上的值与初次检索结果中检查类型，或者性别中的值进行比较，将全匹配的信息显示出来。</p>
    <p>[0068]	4、综合性检索</p>
    <p>[0069]	综合类型检索是以上几种检索方式结合的检索，即几种检索条件共同作用于Li st，显示同时符合几种条件限制的检索信息。</p>
    <p>[0070]	如图10所示，当找到符合条件的信息的时候，选中符合条件的信息(为二次检索结果)，触发影像显示事件，将选中信息对应的影像信息(包括StudyUID、SeriesUID和SOPInstanceUID)组合成WADO协议串，将WADO协议串传送给WADO服务，获取需要的影像。</p>
    <p>[0071]	影像显示即调用DICOM影像。DICOM影像检索的最终的目的就是显示影像。当根据检索找到所需的信息时，点击该信息，触发影像调阅事件，将本条信息中WADO协议串与WADO服务进行交互，提取对应信息的DICOM影像。影像显示的效果图如图11所示，图11的左侧为序列图的列表图框，右侧为显示的选中的序列框的影像。</p>
