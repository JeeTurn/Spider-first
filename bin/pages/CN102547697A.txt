    <p>业务流传输方法、业务平台设备和用户终端技术领域[0001]	本发明涉及通信技术领域，尤其涉及一种业务流传输方法、业务平台设备和用户终端。背景技术[0002]	现有技术中，用户终端（UE，User Equipment)通过2G/时分同步码分多址 (TD-SCDMA,Time Division-Synchronous Code Division Multiple Access)无线接入，并经由无线应用协议(WAP,Wireless Application Protocol)网关（WAPGW)访问 WAP/WWW业务的流程如图1所示：[0003]	步骤11，UE通过2G/TD-SCDMA无线接入至服务GPRS支持节点（SGSN，Servicing GPRS Support Node)和网关 GPRS 支持节点（GGSN，Gateway GPRSSupport Node)后，GGSN 首先向WAP网关（WAP Gff, WAP Gateway)中的!Radius接入请求消息处理单元发送!Radius 接入请求信息（Access Request)，该Access Request中携带有UE的移动用户国际号码 (MSISDN, Mobile SubscriberInternational ISDN)和网络接入类型(RAT Type)；[0004]	步骤12，Radius接入请求消息处理单元在接收到Access Request后，向PDN Gff 发送响应消息Access Acc印ted，表示接收该UE的HTTP访问请求；[0005]	步骤13，GGSN向Radius接入请求消息处理单元发送计费请求消息Accounting Request,该 Accounting Request 中7&#190;载该 UE 的 MSISDN 禾口 RATType ；[0006]	步骤14，Radius接入请求消息处理单元将Accounting Request发送给WAPGW ；[0007]	步骤15，WAP GW向！Radius接入请求消息处理单元发送响应消息Accounting Response,表不成功接收 Accounting Request ；[0008]	步骤16，Radius接入请求消息处理单元将Accounting Response转发给GGSN ；[0009]	步骤17，UE向WAP Gff发送第一业务平台访问请求；[0010]	步骤18，WAP GW将该第一业务平台访问请求发送至第一业务平台（比如可以为移动梦网），该第一业务平台访问请求的x-up-calling-line字段中承载有UE的MSISDN和 RAT Type ；[0011]	步骤19，第一业务平台将相应的业务流反馈给WAP GW；[0012]	步骤110，WAP Gff将相应的业务流反馈给UE ；[0013]	步骤111，UE向WAP GW发送获取业务流的第二业务平台访问请求，比如，可以假设该第二业务平台为QQ业务平台这样的第三方业务平台；[0014]	步骤112，WAP Gff将第二业务平台访问请求发送给第二业务平台，由于QQ业务平台为第三方业务平台，此时的第二业务平台访问请求中无需承载UE的MSISDN，而只需承载 RAT Type ；[0015]	步骤113，第二业务平台将相应的业务流反馈给WAP GW；[0016]	步骤114，第一业务平台将相应的业务流反馈给UE。[0017]	在上述步骤中，WAP GW作为代理网关，会根据UE访问的目的地址判断是否在发送给业务平台的信息中携带MSIDSN。比如，针对中国移动自有业务和合作类业务，如飞信、移动梦网等，WAP网关在向该些业务的业务平台发送访问请求时，会将MSISDN承载在前传给业务平台的访问请求的x-up-calling-line字段中，便于业务平台根据该 x-up-calling-line对UE进行鉴权和计费。[0018]	在现有技术提供的上述流程中，通过WAP GW访问WAP/WWW业务的Access Request 以及业务流都会经过WAP GW。随着时分同步码分多址长期演进（TD-LTE，TD-SCDMA Long Term Evolution)的引入，UE能力得到不断提升，WAP GW将面对更多大流量的业务流传输， 如视频流传输、大包下载等等。UE访问流量的爆炸式增长使得WAP GW成为业务流传输的瓶颈。因此，现有技术中急需一种能够减轻WAP GW承担的业务流传输量的方案。发明内容[0019]	本发明实施例提供一种业务流传输方法、业务平台设备和用户终端，用以解决现有技术中存在的WAP Gff承担的业务流传输量过大的问题。[0020]	本发明实施例采用以下技术方案：[0021]	一种业务流传输方法，包括：业务平台接收无线应用协议网关WAP GW转发的由用户终端发送的第一访问请求；以及生成用于指示所述用户终端鉴权通过的第一鉴权标识， 并将所述第一鉴权标识发送给所述用户终端，指示用户终端直接向所述业务平台发送承载有所述第一鉴权标识的业务流。[0022]	一种业务流传输方法，包括：用户终端通过无线应用协议网关WAP Gff向业务平台发送第一访问请求；并接收所述业务平台发送的用于指示所述用户终端鉴权通过的鉴权标识；以及直接向所述业务平台发送承载有所述鉴权标识的业务流。[0023]	一种业务平台设备，包括：第一访问请求接收单元，用于接收无线应用协议网关 WAP GW转发的由用户终端发送的第一访问请求；鉴权标识生成单元，用于在第一访问请求接收单元接收到所述第一访问请求后，生成用于指示所述用户终端鉴权通过的第一鉴权标识；鉴权标识发送单元，用于将鉴权标识生成单元生成的第一鉴权标识发送给所述用户终端，指示用户终端直接向所述业务平台发送承载有所述第一鉴权标识的业务流。[0024]	一种用户终端，包括：第一访问请求发送单元，用于通过无线应用协议网关WAP Gff 向业务平台发送第一访问请求；鉴权标识接收单元，用于接收所述业务平台在接收到第一访问请求发送单元发送的第一访问请求后，发送的用于指示所述用户终端鉴权通过的鉴权标识；业务流发送单元，用于直接向所述业务平台发送承载有鉴权标识接收单元接收的鉴权标识的业务流。[0025]	本发明实施例的有益效果如下：[0026]	本发明实施例提供的上述方案通过将第一条访问请求（即第一访问请求）外的其他业务流从WAP Gff中剥离开来，直接在用户终端与业务平台之间进行这部分业务流的传输，从而解决了现有技术中存在的WAP GW承担的业务流传输量过大的问题。附图说明[0027]	图1为现有技术中UE访问WAP/WWW业务的流程示意图；[0028]	图2为本发明实施例提供的一种业务流传输方法的具体流程示意图；[0029]	图3为本发明实施例提供的另一种业务流传输方法的具体流程示意图；[0030]	图4为本发明实施例提供的方案在实际中的应用流程示意图；[0031]	图5为本发明实施例提供的一种业务平台设备的具体结构示意图；[0032]	图6为本发明实施例提供的一种用户终端的具体结构示意图。具体实施方式[0033]	为了减轻WAP GW承担的业务流传输量，发明人对现有技术提供的业务流传输方案进行了深入研究。研究表明，类似Access Request这样的请求主要是为了提供UE的MSIDSN 以进行业务鉴权等，这样的请求有必要经过WAPGW ；而在业务鉴权完成后，UE与业务平台之间交互的业务流则没有经过WAP网关的必要性，而这部分业务流流经WAP Gff恰恰是导致 WAP GW负担过重的原因。[0034]	根据上述分析，本发明实施例提供了一种用于减轻WAP Gff的业务流传输量的方案。该方案的主要思想在于，保留与现有技术类似的将Access Request和第一条访问请求经WAP GW提供给业务平台的方式，而将除Access Request和第一条访问请求外的其他业务流从WAP GW中剥离开来，直接在GGSN与业务平台之间进行这部分业务流的传输。[0035]	以下结合附图，对本发明实施例提供的方案进行详细说明。[0036]	首先，本发明实施例提供一种业务流传输方法，该方法的具体流程示意图如图2 所示，包括以下步骤：[0037]	步骤21，业务平台接收WAP GW转发的由用户终端发送的第一访问请求；[0038]	步骤22，由于WAP Gff对第一访问请求进行转发的前提时WAP Gff已对发送该第一访问请求的用户终端进行了鉴权，并鉴权通过，因此，业务平台执行生成用于指示用户终端鉴权通过的第一鉴权标识的操作；[0039]	步骤23，业务平台将该第一鉴权标识发送给用户终端，指示用户终端不经过WAP GW而直接向业务平台发送承载有该第一鉴权标识的业务流。[0040]	通过本发明实施例提供的上述方法，业务平台在接收到用户终端发送的第一访问请求后，向用户终端反馈一个鉴权标识，使用户终端后续可以不经过WAP GW而直接向业务平台发送承载有鉴权标识的业务流，从而一方面满足减轻WAP Gff负载的需求，一方面也能保证业务平台后续能够根据该鉴权标识实现对用户终端的计费等操作。[0041]	本发明实施例提供的上述方法中，业务平台在将第一鉴权标识发送给用户终端后，若接收到用户终端发送的承载有第二鉴权标识的第二访问请求，则为了进一步减轻WAP GW承担的业务流传输量，业务平台可以在比较出第一鉴权标识与第二鉴权标识匹配一致后，确定用户终端为鉴权通过的用户终端，从而不经过WAP GW而直接向用户终端发送与该第二访问请求相应的业务流。[0042]	本发明还提供另一种业务流传输方法，该方法的具体流程示意图如图3所示，包括下述步骤：[0043]	步骤31，用户终端通过无线应用协议网关WAP Gff向业务平台发送第一访问请求；[0044]	步骤32，该用户终端接收业务平台发送的用于指示该用户终端鉴权通过的鉴权标识；[0045]	步骤33，该用户终端不经过WAP Gff而直接向业务平台发送承载有该鉴权标识的业务流。[0046]	在本发明实施例中，用户终端可以在接收到上述鉴权标识后，在第二访问请求中承载有该鉴权标识，采用不经过WAP Gff而直接向业务平台发送的方式进行发送，并接收业务平台不经过WAP Gff而直接反馈的与该第二访问请求相应的业务流。[0047]	以下以本发明实施例提供的上述方案在实际中的应用流程为例，详细说明上述方案的实现原理。具体地，上述方案在实际中的应用流程示意图如图4所示，包括下述步骤：[0048]	步骤41，用户终端UE通过PDN Gff向WAP Gff中Radius接入请求消息处理单元发送一个HTTP访问请求（Access Request)，该Access Request为占用较小流量的数据包，其中承载有MSISDN和RAT Type ；[0049]	步骤42，Radius接入请求消息处理单元在接收到Access Request后，向PDN Gff 发送响应消息Access Acc印ted，表示接收该UE的HTTP访问请求；[0050]	步骤43，PDN Gff向Radius接入请求消息处理单元发送计费请求消息Accounting Request,该 Accounting Request 中7&#190;载该 UE 的 MSISDN 禾口 RATType ；[0051]	步骤44，Radius接入请求消息处理单元将Accounting Request发送给WAPGW ；[0052]	步骤45，WAP Gff向！Radius接入请求消息处理单元发送响应消息Accounting Response,表不成功接收 Accounting Request ；[0053]	步骤46，Radius接入请求消息处理单元将Accounting Response转发给PDNGW ；[0054]	步骤47，UE向WAP Gff发送第一个HTTP访问请求；[0055]	步骤48，WAP GW按照与现有技术类似的处理流程，从HTTP访问请求中获取 MSISDN,并将其承载到x-up-calling-line字段中后，将包含该x-up-calling-line字段的 HTTP访问请求前传给业务平台；[0056]	步骤49，业务平台在接收到该第一个HTTP访问请求后，从该访问请求中获取UE的 MSISDN,并针对该UE，生成一个用于标识该UE的唯一随机数token，作为后续业务流访问的鉴权标识；[0057]	步骤410，业务平台在返回给WAP Gff的响应消息中承载该token ；[0058]	步骤411，WAP Gff将该响应消息发送给UE ；[0059]	步骤412，UE在接收到响应消息后，根据响应消息中承载的token，执行业务流访问配置重设置的操作，执行该操作的目的在于使后续UE与业务平台之间交互的业务流不再发送给WAP Gff,而是直接在UE与业务平台之间进行交互；[0060]	步骤413，UE向PDN Gff发送承载有token的业务访问请求； [0061 ] 步骤414，PDN Gff将该业务访问请求发送给业务平台；[0062]	步骤415，业务平台判断分配给各UE的token中是否存在该业务访问请求中承载的token，并在确定结果为是时，确定该UE鉴权通过；[0063]	步骤416，业务平台向PDN Gff发送鉴权通过响应消息；[0064]	步骤417，PDN Gff将鉴权通过响应消息发送给UE。[0065]	在上述步骤中，Serving Gff的功能类似于2G/TD-SCDMA下的SGSN，而PDN Gff的功能类似于2G/TD-SCDMA下的GGSN0[0066]	对应于本发明实施例提供的业务流传输方法，本发明实施例还提供一种业务平台设备，用以解决现有技术中存在的WAP GW承担的业务流传输量过大的问题。该业务平台设备的具体结构示意图如图5所示，包括以下功能单元：[0067]	第一访问请求接收单元51，用于接收WAP GW转发的由用户终端发送的第一访问请求；[0068]	鉴权标识生成单元52，用于在第一访问请求接收单元51接收到第一访问请求后， 生成用于指示用户终端鉴权通过的第一鉴权标识；[0069]	鉴权标识发送单元53，用于将鉴权标识生成单元52生成的第一鉴权标识发送给用户终端，指示用户终端直接向业务平台发送承载有第一鉴权标识的业务流。[0070]	在一个较佳的实施例中，本发明实施例提供的上述业务平台设备还可以进一步包括：[0071]	第二访问请求接收单元，用于在鉴权标识发送单元53将第一鉴权标识发送给用户终端后，接收用户终端发送的承载有第二鉴权标识的第二访问请求；[0072]	比较单元，用于比较第一鉴权标识与第二访问请求接收单元接收的第二访问请求中承载的第二鉴权标识是否匹配一致；[0073]	业务流发送单元，用于在比较单元得到的比较结果为是时，直接向用户终端发送与第二访问请求相应的业务流。[0074]	对应于本发明实施例提供的业务流传输方法，本发明实施例还提供一种用户终端，用以解决现有技术中存在的WAP GW承担的业务流传输量过大的问题。该用户终端的具体结构示意图如图6所示，包括以下功能单元：[0075]	第一访问请求发送单元61，用于通过无线应用协议网关WAP GW向业务平台发送第一访问请求；[0076]	鉴权标识接收单元62，用于接收业务平台在接收到第一访问请求发送单元61发送的第一访问请求后，发送的用于指示用户终端鉴权通过的鉴权标识；[0077]	业务流发送单元63，用于直接向业务平台发送承载有鉴权标识接收单元62接收的鉴权标识的业务流。[0078]	在一个较佳的实施例中，当业务流发送单元63发送的是承载有鉴权标识的第二访问请求时，该用户终端还可以进一步包括：[0079]	业务流接收单元，用于在业务流发送单元63向业务平台发送第二访问请求后，接收业务平台直接向用户终端发送的与第二访问请求相应的业务流。[0080]	显然，本领域的技术人员可以对本发明进行各种改动和变型而不脱离本发明的精神和范围。这样，倘若本发明的这些修改和变型属于本发明权利要求及其等同技术的范围之内，则本发明也意图包含这些改动和变型在内。</p>
