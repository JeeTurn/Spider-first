    <div class="abstract">本发明公开了一种指示业务类型的方法、装置及系统，用以对日益复杂的各类应用业务进行明确分类，避免业务处理流程发生混淆。该方法为：核心网设备通过下行报文中的GTP-U信元内的扩展头将指示业务类型的业务标识发给接入网设备，令接入网设备可以及时确定业务类型并执行相应的业务处理操作，显然，通过GTP-U信元内的扩展头来携带业务标识，可以支持更多种类的业务类型，有利于业务类型的扩展，同时，由于GTP-U信元不属于IP传输层的层面，解析时不涉及底层传输层面的修改，因此，采用GTP-U信元携带业务标识不会涉及系统的大范围改造，从而有效节省了实现成本，也避免了给系统性能造成重大影响，保证了系统的服务质量。</div>

    <p>一种指示业务类型的方法、装置及系统</p>
    <p>技术领域</p>
    <p>[0001]	本发明涉及通信领域，特别涉及一种指示业务类型的方法、装置及系统。 背景技术</p>
    <p>[0002]	目前，在 TD-SCDMA(Time Division-Synchronous Code Division Multiple Access，时分同步码分多址）系统中，只能通过在建立业务时采用的RAB(Radio Access Bearer，无线接入承载）指派消息中的消息类型（会话类、流类、交互类、背景类）和QoS参数简单对业务类型进行分类，然后，再根据业务类型进行不同的无线资源调度。随着3G的规模组网，业务类型也更加丰富，针对不同业务类型需要有不同的无线资源调度策略。现有的业务分类方法已经不能满足运营商对业务分类的要求。</p>
    <p>[0003]	为了解决上述问题，现有技术下，核心网设备通常通过在传输下行数据时使用下行报文的IP头中的DSCP (Differentiated Services Code Poin，区分服务代码属性）参数来指示业务类型，例如，核心网设备识别出当前待处理业务的业务类型后，根据预设规则填写下行报文的IP头中的DSCP参数，即采用某一特定的DSCP值对应表示某一种业务类型， DSCP参数在下行报文内的IP头中的位置具体参阅图1所示。如图1所示，现有技术下，IP 头中的DSCP参数仅包含一个字节，即8个bits，取值范围有限，且其中很多取值均已被使用，不能再作为业务类型的标识，因此，不利于业务类型的扩展；同时，由于DSCP参数处于 IP传输层的层面，解析时涉及底层传输层面的修改，因此，若修改DSCP参数占用的字节数以支持其表征更多的业务类型，则涉及改动的范围较大，会对设备性能造成重大影响。</p>
    <p>[0004]	有鉴于此，需要重新设计一种指示业务类型的方法，以克服上述缺陷。</p>
    <p>发明内容</p>
    <p>[0005]	本发明实施例提供一种指示业务类型的方法、装置及系统，用以对日益复杂的各类应用业务进行明确分类，避免业务处理流程发生混淆。</p>
    <p>[0006]	本发明实施例提供的具体技术方案如下：</p>
    <p>[0007]	&#8212;种指示业务类型的方法，其特征在于，包括：</p>
    <p>[0008]	核心网设备确定当前待处理业务的业务类型；</p>
    <p>[0009]	核心网设备将所述业务类型对应的业务标识添加在该业务的下行报文的GTP-U 信元中；</p>
    <p>[0010]	核心网设备将所述下行报文发往接入网设备，以通过所述下行报文携带的GTP-U 信元将当前待处理业务的业务类型通知所述接入网设备。</p>
    <p>[0011]	一种进行业务处理的方法，包括：</p>
    <p>[0012]	接入网设备接收核心网设备发送的下行报文，该下行报文的GTP-U信元中携带有当前待处理业务的业务类型对应的业务标识。</p>
    <p>[0013]	接入网设备对接收的下行报文进行解析，并根据所述下行报文的GTP-U信元中携带的业务标识确定当前待处理业务的业务类型；[0014]	接入网设备获取对应当前待处理业务的业务类型预设的业务处理策略，并采用该业务处理策略对当前待处理业务进行处理。</p>
    <p>[0015]	 一种指示业务类型的装置，包括：</p>
    <p>[0016]	确定单元，用于确定当前待处理业务的业务类型；</p>
    <p>[0017]	处理单元，用于将所述业务类型对应的业务标识添加在该业务的下行报文的 GTP-U信元中；</p>
    <p>[0018]	通信单元，用于将所述下行报文发往接入网设备，以通过所述下行报文携带的 GTP-U信元将当前待处理业务的业务类型通知所述接入网设备。</p>

    <p>[0021]	解析单元，用于对接收的下行报文进行解析，并根据所述下行报文的GTP-U信元中携带的业务标识确定当前待处理业务的业务类型；</p>
    <p>[0022]	处理单元，用于获取对应当前待处理业务的业务类型预设的业务处理策略，并采用该业务处理策略对当前待处理业务进行处理。</p>
    <p>[0023]	一种用于指示业务类型及业务处理的系统，包括：</p>
    <p>[0024]	核心网设备，用于确定当前待处理业务的业务类型，并将所述业务类型对应的业务标识添加在下行报文的GTP-U信元中，以及将所述下行报文发往接入网设备，以通过所述下行报文携带的GTP-U信元将当前待处理业务的业务类型通知所述接入网设备；</p>
    <p>[0025]	接入网设备，用于接收核心网设备发送的下行报文，该下行报文的GTP-U信元中携带有当前待处理业务的业务类型对应的业务标识，对接收的下行报文进行解析，并根据所述下行报文的GTP-U信元中携带的业务标识确定当前待处理业务的业务类型，以及获取对应当前待处理业务的业务类型预设的业务处理策略，并采用该业务处理策略对当前待处理业务进行处理。</p>
    <p>[0026]	本发明实施例中，核心网设备通过下行报文中的GTP-U信元内的扩展头将指示业务类型的业务标识发给接入网设备，令接入网设备可以及时确定业务类型并执行相应的业务处理操作，显然，通过GTP-U信元内的扩展头来携带业务标识，可以支持更多种类的业务类型，有利于业务类型的扩展，同时，由于GTP-U信元不属于IP传输层的层面，解析时不涉及底层传输层面的修改，因此，采用GTP-U信元携带业务标识不会涉及系统的大范围改造， 从而有效节省了实现成本，也避免了给系统性能造成重大影响，保证了系统的服务质量。</p>

    <p>[0034]	本发明实施例提供一种指示业务类型的方法、装置及系统，用以对日益复杂的各类应用业务进行明确分类，避免业务处理流程发生混淆。具体为：核心网设备在传输下行数据报 （简称下行报文）时，使用下行报文中的GTP-U信元来指示当前待处理业务的业务类型，例如，核心网设备识别出当前待处理业务的业务类型后，根据预配置规则填写GTP-U 信元中扩展头的业务类型扩展字段，即用某一特定的取值对应表示某一种业务类型。核心网设备将携带有上述GTP-U信元的下行报文传送给接入网设备后，RNC便可以根据接收的下行报文中的GTP-U信元携带的用于指示业务类型的业务标识以及预配置规则，确定出当前待处理业务的业务类型，从而执行相应的无线资源调度策略。</p>
    <p>[0035]	实际应用中，所谓的待处理业务即是指核心网设备需要传输的每一包数据，在针对每一包数据发送的下行报文中均需要携带相应的业务标识，从而令接入网设备可以识别出每一包数据的业务类型，以采用相适应的业务处理策略。</p>

    <p>[0038]	核心网设备，用于确定当前待处理业务的业务类型，并将所述业务类型对应的业务标识添加在下行报文的GTP-U信元中，以及将下行报文发往接入网设备，以通过该下行报文携带的GTP-U信元将当前待处理业务的业务类型通知接入网设备；</p>
    <p>[0039]	接入网设备，用于接收核心网设备发送的下行报文，该下行报文的GTP-U信元中携带有当前待处理业务的业务类型对应的业务标识，对接收的下行报文进行解析，并根据所述下行报文的GTP-U信元中携带的业务标识确定当前待处理业务的业务类型，以及获取对应当前待处理业务的业务类型预设的业务处理策略，并采用该业务处理策略对当前待处理业务进行处理。</p>

    <p>[0046]	通信单元40，用于接收核心网设备发送的下行报文，该下行报文的GTP-U信元中携带有当前待处理业务的业务类型对应的业务标识。</p>
    <p>[0047]	解析单元41，用于对接收的下行报文进行解析，并根据所述下行报文的GTP-U信元中携带的业务标识确定当前待处理业务的业务类型；</p>
    <p>[0048]	处理单元42，用于获取对应当前待处理业务的业务类型预设的业务处理策略，并采用该业务处理策略对当前待处理业务进行处理。</p>
    <p>[0049]	参阅图5A所示，本发明实施例中，核心网设备向接入网设备指示业务类型的详细流程如下：</p>
    <p>   [0050]	步骤500 ：核心网设备确定当前待处理业务的业务类型。</p>
    <p>[0051]	本发明实施例中，核心网设备可以根据用户请求的具体业务内容确定当前待处理业务的业务类型。</p>
    <p>[0052]	步骤510 ：核心网设备将当前待处理业务的业务类型对应的业务标识添加在该业务的下行报文的GTP-U信元中。</p>
    <p>[0053]	本实施例中，在业务类型和业务标识之间设置有一定的映射关系，例如：例如： Oxll表示QQ业务，0x12表示MSN业务，0x01表示手机支付业务、手机证券业务，0x05表示下载业务。</p>
    <p>[0054]	核心网设备即是将对应业务类型预设的业务标识添加到下行报文的GTP-U信元中。</p>
    <p>[0055]	进一步地，本实施例中，核心网设备在GTP-U信元中添加业务类型的业务标识时， 将业务类型的业务标识添加在GTP-U信元的扩展头中，该扩展头为本实施例中新定义的扩展头，扩展头的字段大小可扩展，取值范围可以根据具体应用环境自行设置，因而可以支持表征大量的业务类型，从而利于业务类型的扩展。本实施例中，GTP-U信元所在下行报文采用的协议栈格式如图5B和图5C所示，其中，图5B ATM (Asynchronous Transfer Mode，异步传输模式)下的下行报文协议栈格式，图5C为IPdnternet Protocol,因特尔协议)传输方式下的下载报文协议栈格式。</p>
    <p>[0056]	参阅图5B和图5C所示，核心网设备可以在下行报文中的GTP-U信元中的协议头的位置填写如表1所示的GTP-U信元标识，用于表示当前下行报文中的GTP-U信元中是否添加了扩展头，如表1所示，若Octets-12对应的字段填写的内容为“Next Extension Header Type3) 4) ”，则说明GTP信元中包含的扩展头。如图5B和图5C所示，在下行报文中， 越上层的协议相应的数据头越在数据的外层，其他上层数据头和数据对于该层来说都是数据静荷，因此，IP层的协议头比GTP-U层的协议头更靠近真正的数据静荷，即更接近底层数据，因此，对IP层的协议进行扩展，其难度要高于对GTP-U层的协议进行扩展，因此，本实施例中，对GTP-U信元进行扩展，令其携带业务标识，可以避免对IP层的协议进行修改，从而避免了对IP层的传输造成较大影响，节省了实现成本，降低了实现难度。</p>
    <p>[0057]	本实施例中，核心网设备在“Next Extension Header Type3)4) ”中可以填写的内容如表2所示：例如，若填写的内容为00100000，则表示在扩展头中添加的内容是Service Class indicator (业务类型指示），即业务标识；若填写的内容为11000000，则表示扩展头中添加的内容是PDCP数据单元个数，......以此类推，不再赘述。</p>
    <p>[0058]	将“Next Extension Header Type3)4) ”填写完毕后，便可以在扩展头中添加业务标识，进一步地，还可以添加其他指定的配置参数，如，参阅表3所示，添加的业务标识为： Service Class indicator (0x11 for QQ)，其他配置参数为：I-(Length)、0xFF_ (dummy)和 00。[0059]	表 1</p>

    <p>[0066]	步骤520 ：核心网设备将下行报文发往接入网设备，以通过该下行报文携带的 GTP-U信元将当前待处理业务的业务类型通知接入网设备</p>
    <p>[0067]	参阅图5B所示，本实施例中，核心网设备之所以采用GTP-U信元将业务类型通知接入网设备，是因为GTP-U相对于IP层更靠近数据静荷，因此，对GTP-U信元进行解析时涉及改动的部分较少，对设备性能影响相较于DSCP参数的解析更小。</p>
    <p>[0068]	基于上述实施例，参阅图6所示，接入网设备根据核心网设备指示的业务类型进行业务处理的详细流程如下：</p>
    <p>[0069]	步骤600 ：接入网设备接收核心网设备发送的下行报文，该下行报文的GTP-U信元中携带有当前待处理业务的业务类型对应的业务标识。</p>
    <p>[0070]	步骤610 ：接入网设备对接收的下行报文进行解析，并根据该下行报文的GTP-U信元中携带的业务标识确定当前待处理业务的业务类型。</p>
    <p>[0071]	本实施例中，结合表1、表2和表3可知，接入网设备在执行步骤610时，是根据下行报文的GTP-U信元的扩展头中携带的业务标识确定当前待处理业务的业务类型的，具体为：接入网设备确定GTP-U信元的协议头位置携带有用于表征GTP-U信元中存在表示业务类型的扩展 头的信息后，在该扩展头中获取用于指示业务类型的业务标识；进一步地，接入网设备还可以在上述扩展头中获取用于指示业务类型的业务标识后，在扩展头中指定的配置参数。</p>
    <p>[0072]	步骤620 ：接入网设备获取对应当前待处理业务的业务类型预设的业务处理策略，并采用该业务处理策略对当前待处理业务进行处理。</p>
    <p>[0073]	本实施例中，针对不同类型的业务预设了不同的业务处理优先级，这个业务优先级即可以看作是业务处理策略，不同业务类型的待处理业务对应不同的业务处理优先级。</p>
    <p>[0074]	接入网设备获取对应当前待处理业务预设的业务处理优先级后，可以根据所述业务处理优先级确定相应的执行策略，并根据该执行策略对当前待处理业务进行处理，所谓的执行策略包含以下参数中的一种或任意组合（即为举例，执行策略可以根据具体应用环境自行设定）：</p>
    <p>[0075]	资源调度优先级：资源调度优先级高的业务优先调度；</p>
    <p>[0076]	资源抢占优先级：资源抢占优先级越低的业务越容易被其他业务抢占无线资源；</p>
    <p>[0077]	默认接入速率，业务处理优先级越高的业务默认接入速率越高。</p>
    <p>[0078]	例如：手机支付业务、手机证券业务可设置最高的业务处理优先级5，下载业务设置中等的业务处理优先级3，QQ和MSN业务设置最低的业务处理优先级1。取值越大，优先级越高。</p>
    <p>[0079]	业务处理优先级为5的手机支付业务、手机证券业务调度时优先于下载业务，在空口具有最大的流量比例（如空口总流量的50%)；下载业务调度时优先于QQ、MSN业务， 具有中等大小的流量比例（如空口总流量的30%) ；QQ、MSN业务的调度优先级最低，只能获得最小的流量比例（如空口总流量的20% )。</p>
    <p>[0080]	对于业务处理优先级为5的手机支付业务、手机证券业务在网络没有资源的情况下可以抢占业务处理优先级为1的QQ、MSN业务的资源和业务处理优先级为3的下载业务的资料，如，先抢QQ、MSN业务的资源，还不能接入再抢下载业务的资源。</p>
    <p>[0081]	在业务接入时，业务处理优先级为5的手机支付业务、手机证券业务可以分配较高的接入速率，如，64k ；业务处理优先级为3的下载业务分配中等的接入速率，如，32k ；而业务处理优先级为1的QQ、MSN业务分配最低的接入速率，如，16k。这样，使得业务处理优先级的用户获得更高的速率，从而有效保证了系统的服务性能。</p>
    <p>[0082]	采用上述技术方案便可以根据业务处理优先级实现对业务的包调度，即通过解析每包数据（即每一条下行报文）的业务类型，在相同的RAB(无线接入承载）上实际不同业务的承载，并且针对每包数据根据其业务类型的不同执行不同优先级的业务处理策略。</p>
    <p>  [0083]	综上所述，本发明实施例中，核心网设备通过下行报文中的GTP-U信元内的扩展头将指示业务类型的业务标识发给接入网设备，令接入网设备可以及时确定业务类型并执行相应的业务处理操作，显然，通过GTP-U信元内的扩展头来携带业务标识，可以支持更多种类的业务类型，有利于业务类型的扩展，同时，由于GTP-U信元不属于IP传输层的层面， 解析时不涉及底层传输层面的修改，因此，采用GTP-U信元携带业务标识不会涉及系统的大范围改造，从而有效节省了实现成本，也避免了给系统性能造成重大影响，保证了系统的服务质量。</p>
    <p>[0084]	显然，本领域的技术人员可以对本发明进行各种改动和变型而不脱离本发明的精神和范围。这样，倘若本发明的这些修改和变型属于本发明权利要求及其等同技术的范围之内，则本发明也意图包含这些改动和变型在内。</p>
