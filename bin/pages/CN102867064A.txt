    <p>关联字段查询装置和关联字段查询方法</p>
    <p>技术领域</p>
    <p>[0001]	本发明涉及计算机技术领域，具体而言，涉及一种关联字段查询装置和一种关联字段查询方法。</p>
    <p>背景技术</p>
    <p>[0002]	随着计算机技术的发展，计算机的主要应用已从传统的科学计算转变为事务数据处理。在事务处理过程中，并不需要复杂的科学计算，而是需要进行大量数据的存储、查找、统计等工作，例如教学管理、人事管理、财务管理等。这需要对大量数据进行管理，数据库技术就是目前最先进的数据定理技术。任何一个数据库管理系统都是基于某种数据模型的，目前常用的数据模型有三种：层次模型、网状模型和关系模型。与之相对应，数 据库也分为三种基本类型：层次型数据库、网状型数据库和关系型数据库。关系模型用二维表结构来表示实体及其之间的联系。关系数据模型以关系数学理论为基础，一个二维表就是一个关系，不仅能描述实体本身，还能反映实体之间的联系。该模型简单，使用方便，应用也最广泛。</p>
    <p>[0003]	由关系模型构成的数据库就是关系数据库。关系数据库由包含数据记录的多个数据表组成，用户可在有相关数据的多个表之间建立相互联系。</p>
    <p>[0004]	在关系数据库中，数据被分散到不同的数据表中，以便使每一个表中的数据只记录一次，从而避免数据的重复输入，减少冗余。</p>
    <p>[0005]	基于关系型数据库的应用软件中存在大量的关联数据查询。但是，当关联字段越来越多时，由于数据库的索引长度存在限制，当索引列的长度超出限制后，无法为数据表创建过多索引，数据库被迫选择全表扫描的方式查询数据，随着数据量的增加，应用系统的处理效率也随之急剧下降。为此，对于此种场景，提出适当的处理方法提高数据库的查询效率是有价值的。</p>
    <p>[0006]	相关技术例如使用计算列来查询关联字段，但是数据库的索引长度存在限制，当索引列的长度超出限制后，将无法为该数据列创建索引。当不能有效利用索引时，数据库系统通过全表扫描查找数据，效率较差，显然这种方案无法有效解决关联字段过多对系统性能产生的负面影响。</p>
    <p>[0007]	因此，需要一种关联字段查询技术，可减少数据查询的字段数量，压缩字段长度，突破数据库索引键值长度和索引列数量的限制，有效利用索引提高数据查询效率。</p>

    <p>[0009]	有鉴于此，根据本发明的一个方面，提供了一种关联字段查询装置，包括：获取单元，获取关系型数据库中数据表之间的关联字段；计算列生成单元，将所述关联字段组合成计算列；摘要生成单元，根据所述计算列计算得到唯一消息摘要；查询单元，根据所述唯一消息摘要创建计算列索引，依据所述计算列索引查询目标数据。</p>

    <p>[0051]	I.建立Bizkey计算列</p>
    <p>[0052]	如图3所不，A、B两数据表的关联字段为Coll&#183;&#183;&#183;ColN,使用特殊字符作为分隔符，将上述多个关联字段组合成计算列bizkey，例如bizkey=C0ll+’ ■’ +Col2+’ ■’ +…+ColN，其中，’ ■’为特殊字符。将多字段关联SQL转换为单一列关联SQL，即将A join Bon A. coll=B. coll and A. col2=B. col2 and …and A. colN=B. colM 简化为 A join B onA. bizkey=B. bizkey。</p>
    <p>[0053]	2.使用消息摘要算法生成Bizkey唯一指纹</p>
    <p>[0054]	消息摘要算法的主要特征是加密过程不需要密钥，并且经过加密的数据无法被解密，只有输入相同的明文数据经过相同的消息摘要算法才能得到相同的密文。消息摘要算法不存在密钥的管理与分发问题，适合于分布式网络相同上使用。由于其加密计算的工作量相当可观，所以以前的这种算法通常只用于数据量有限的情况下的加密，例如计算机的口令就是用不可逆加密算法加密的。近年来，随着计算机相同性能的飞速改善，加密速度不再成为限制这种加密技术发展的桎梏，因而消息摘要算法应用的领域不断增加。</p>
    <p>[0055]	现在，消息摘要算法主要应用在“数字签名”领域，作为对明文的摘要算法。著名的摘要算法有MD5 (信息摘要算法)、SHA-U RIPEMD (原始完整性校验消息摘要)、Haval等算法及其大量的变体。</p>
    <p>[0056]	消息摘要是把任意长度的输入糅合而产生长度固定的伪随机输入的算法。消息摘要的主要特点有：</p>
    <p>[0057]	①无论输入的消息有多长，计算出来的消息摘要的长度总是固定的。例如应用MD5算法摘要的消息有128个比特位，用SHA-I算法摘要的消息最终有160比特位的输出，SHA-I的变体可以产生192比特位和256比特位的消息摘要。一般认为，摘要的最终输出越长，该摘要算法就越安全。</p>
    <p>[0058]	②消息摘要看起来是“随机的”。这些比特看上去是胡乱的杂凑在一起的。可以用大量的输入来检验其输出是否相同，一般，不同的输入会有不同的输出，而且输出的摘要消息可以通过随机性检验。但是，一个摘要并不是真正随机的，因为用相同的算法对相同的消息求两次摘要，其结果必然相同；而若是真正随机的，则无论如何都是无法重现的。因此消息摘要是“伪随机的”。</p>
    <p>[0059]	③一般地，只要输入的消息不同，对其进行摘要以后产生的摘要消息也必不相同；但相同的输入必会产生相同的输出。这正是好的消息摘要算法所具有的性质：输入改变了，输出也就改变了 ；两条相似的消息的摘要确不相近，甚至会大相径庭。</p>
    <p>[0060]	④消息摘要函数是无陷门的单向函数，即只能进行正向的信息摘要，而无法从摘要中恢复出任何的消息，甚至根本就找不到任何与原信息相关的信息。当然，可以采用强力  攻击的方法，即尝试每一个可能的信息，计算其摘要，看看是否与已有的摘要相同，如果这样做，最终肯定会恢复出摘要的消息。但实际上，要得到的信息可能是无穷个消息之一，所以这种强力攻击几乎是无效的。</p>
    <p>[0061]	⑤好的摘要算法，没有人能从中找到“碰撞”，虽然“碰撞”是肯定存在的。即对于给定的一个摘要，不可能找到一条信息使其摘要正好是给定的。或者说，无法找到两条消息，是它们的摘要相同。</p>
    <p>[0062]	在本实施例中，主要利用上述摘要算法①②③⑤特点中的几个特性：生成的密文长度固定、密文的唯一性(不同输入密文不同，相同输入密文相同）、无碰撞发生，通过MD5、SHA-U RIPEMD或Haval等算法，将任意长度的bizkey计算列转换为一个固定长度的字符串，相当于生成一个唯一的“数字指纹”。使用特殊字符作为分隔符，既保证了字符串的唯一性，同时又避免散列碰撞的发生。</p>
    <p>[0063]目前所有关系型数据库对于索引键的长度、索引列的个数均有限制，如SQLServer索引键最大长度限制为900字节、Oracle索引键长度限制为（db_block_size) X 40%字节、DB2索引键长度限制为2000字节。一旦超出索引键长度限制或列数限制，将无法创建索引，导致数据库只能使用全表扫描的方式查询数据，性能大幅降低。</p>
    <p>[0064]	以基于MD5算法实现压缩bizkey计算列为例，在数据库和应用服务中分别实现相同的代码逻辑，确保在代码端和数据库端执行MD5加密时，对于相同的输入生成相同的128位固定长度密文。</p>
    <p>[0065]	例如假设A数据表的数据如图4所示，已根据关联字段得到计算列（bizkey)。</p>
    <p>[0066]	步骤I :定义bizkey规则。</p>
    <p>[0067]	Bizkey=coll+，_，+col2+，_，+col3+，_，+col4+，_，+col5+，_，+col6+，_，+co17+，If +col8+，If +col9+，If +col 10o</p>
    <p>[0068]	步骤2 :调用getMD5Key得到压缩后的128位密文：</p>
    <p>[0069]	Select F_GetMD5Key (bizkey) from TableA,该密文如图 5 所不。</p>
    <p>[0070]	步骤3 :用128位密文替换原始的bizkey计算列，A数据表（TableA)的数据如图6所示。[0071]	Bizkey字段的长度始终为32字节。简化步骤1、2、3，即定义bizkey规则为：</p>
    <p>[0072]	Bizkey=F&#8212;GetMD5key (coll+，_，+col2+，_，+col3+，_，+col4+，_，+col5+，■，+col6+，■’ +col7+，■’ +col8+，■’ +col9+，■’ +collO)。</p>
    <p>[0073]	3.建立MD5Bizkey计算列索引</p>
    <p>[0074]	将多字段关联转换为单一的md5Bizkey列关联后，数据列的长度也限制在了 128位，不会再引发索引键长度限制问题。因此，无论关联的字段再多、存储的数据再长，都可以通过该方法简化SQL关联查询，对MD5Bizkey列建立索引，通过索引查找加速查询效率。</p>
    <p>[0075]创建	MD5Bizkey 列索引：</p>
    <p>[0076]	Create Index IX&#8212;Bizkey on TableA(Bizkey) 0 [0077]	假设TableA为数据源，且数据量较大，TableB为驱动表且数据量较小，根据B表数据关联查询A表数据，查询多个纬度多字段关联数据时，SQL及查询计划如图7所示。</p>
    <p>[0078]	因此，与全表扫描不同，使用本发明创建的md5bizkey计算列索引可以快速查找到A表中满足条件的记录。</p>
    <p>[0079]	当需要查询特定目标的数据时，如coll = a，col2=b，col3=c，col4=d…coll0=I，通过代码端与数据库端生成的密文匹配即可：</p>
    <p>[0080]	Select * from TableA where bizkey=F&#8212;GetMD5key (coll+，_，+col2+，_，+c�13+，■，+col4+’ ■，+col5+’ ■，+col6+’ ■，+col7+’ ■，+col8+’ ■，+col9+’ ■，+collO)。</p>
    <p>[0081]	下面以测试数据说明此本发明的技术方案对于多字段关联查询的效率提升的情况。</p>
    <p>[0082]	1.首先创建查询的数据源A表</p>
    <p>[0083]	Create table A(coll nvarchar(max), col2 nvarchar(max), col3nvarchar(max)</p>
    <p>，col4 nvarchar (max)，co15 nvarchar(max))</p>
    <p>[0084]	declare@i int</p>
    <p>[0085]	set@i=0</p>
    <p>[0086]	declare@content nvarchar (max)</p>
    <p>[0087]&#8212;插入的&#169;content数据长度超出最大索引键的限制</p>
    <p>[0088]	set</p>
    <p>[0089]	&#169;content=’ abcdefghi jklmnopqrstuvwxyzabcdefghi jklmnopqrstuvwxyzabcdehijklmnopqrstuvwxyzabcdefghijklmnopqrstuvwxyzabcdeffghijklmnopqrstuvwxyzabcdefghijklmnop......</p>
    <p>[0090]-插入I万行测试数据。</p>
    <p>[0091]	2.创建驱动表B</p>
    <p>[0092]	Create table B (co 11 varchar (max) , co I 2 nvarchar (max) , co I 3nvarchar(max)，col4 nvarchar(max)，col5 nvarchar (max))</p>
    <p>[0093]&#8212;插入10条数据，根据B表的10条数据关联查询A表中匹配的数据。</p>
    <p>[0094]	3.原有方案查询</p>
    <p>[0095]	Select &#27706; from A inner join B on a. coll=b. coll and a. col2=b. col2 anda. col3=b. col3 and a. col4=b. col4 and a. col5=b. col5。</p>
    <p>[0096]&#8212;无法创建C0ll+C012+C0l3+C0l4+C0l5上的索引，超出索引键长度限制。[0097]	create index IX_idx on A(coll, col2, col3, co4, col5),创建索引失败。</p>
    <p>[0098]	使用全表扫描查询，查询过程参见图8。全表扫描10000条数据，查找到符合条件的记录，消耗很高。查询耗时27. 2秒，CPU消耗21949，逻辑读Reads为2608003。</p>
    <p>[0099]	4.新方案查询</p>
    <p>[0100]&#8212;建立Bizkey计算列：</p>
    <p>[0101]	alter table A add md5bizkey nvarchar (1000)；</p>
    <p>[0102]	alter table B add md5bizkey nvarchar (1000)。</p>
    <p>[0103]&#8212;使用MD5算法生成128位密文： </p>
    <p>[0104]	Update A set md5bizkey=f_getmd5key([coll]+' _，+[col2]+，_，+[col3]+，■’+ [col4]+’ ■ ’+[col5])；</p>
    <p>[0105]	Update B set md5bizkey=f_getmd5key([coll]+' _，+[col2]+，_，+[col3]+，■’+ [col4]+’ _’+[col5])。</p>
    <p>[0106]&#8212;建立MD5Bizkey计算列索引：</p>
    <p>[0107]	Create index IX_MD5Bizkey on A(md5bizkey)。</p>
    <p>[0108]-多字段关联查询SQL简化如下：</p>
    <p>[0109]	select a.&#27706;from A inner join B on a. md5bizkey=b. md5bizkey。</p>
    <p>[0110]	使用md5Bizkey索引查找,通过md5bizkey计算列索引，避免了全表扫描，快速定位10条数据。对比可以发现，执行时间从原方案的27. 2秒，下降到0. 2秒，查询效率提升136倍，CPU开销从21949下降到16，逻辑读Reads从2608003下降到329，整体开销均下降几个数量级。</p>
    <p>[0111]	因此，根据本发明的技术方案解决了多维数据的快速匹配问题，特别是当关联字段的维度增加到一定数量级以后，通过MD5压缩加密生成多维度计算列的唯一指纹，大幅提高了通过数据库快速查找到准确的信息的查询效率。</p>
    <p>[0112]	以上所述仅为本发明的优选实施例而已，并不用于限制本发明，对于本领域的技术人员来说，本发明可以有各种更改和变化。凡在本发明的精神和原则之内，所作的任何修改、等同替换、改进等，均应包含在本发明的保护范围之内。</p>
